# EVOLUTION — luca 的设计史

> luca 自己也是个产品，会迭代、会踩坑、会沉淀。这里记**结构性**演进——不只"改了什么"，更是"为什么改、考虑过什么、可复用的 insight"。
> 两节：**沉淀的方法论**（顶部，跨项目的财富，原则）+ **演进日志**（下方，具体事件）。每次对 luca 做结构性改动，在日志加一条，把可复用的 insight 提到方法论。

## 自我精简（最高一条，约束这份文件本身）

**模型变强，harness 该变薄。** luca 的每条机制都编码着一个"模型自己做不到"的假设。改 luca 前先问一句：**这条机制防的失败，现在还会发生吗？** 不会了就**删**，别只加不减。

机制烂掉的方式从来不是被推翻，是**被绕开后还躺在文档里装有效**。**简单是赢来的、得有人守**——这份文件就是守它的地方：任何"给 luca 加个 X"的冲动，先过这一关。

## 沉淀的方法论

1. **工具 vs 助手 = 谁吸收复杂性**。工具把复杂性外包给人（填表、走流程、维护 schema）；助手把复杂性吸进自己，人只**扔料、说要干什么、对递上来的拍板**。luca 走后者。
2. **有 AI 参与的系统要有灵魂**。它与人是**合作**关系，不是人与工具——否则浪费 AI 的思维。灵魂（`SOUL.md` 人格）和肉体（`CLAUDE.md` / `skills/` / 账本）分两层养、分开迭代。
3. **配合默认、主动例外**。主动只在两种触发破例（你要求 / 合作需要吭一声）；"我能多做所以多做"是工具的脸，不是搭档的。
4. **灵魂决定肉体，不是反过来**。每要造一处肉体（目录 / 机制 / skill），先问"哪个伙伴行为伸手要它"——没人要的别造（投机性身体 = 又一次开发者本能赢了）。
5. **伙伴罗盘（防角色绑定）**。每个人按自己的工作模式造东西（程序员 → 工具，PM → PDM）。要不被"开发者造工具"的引力拽走，就问"**一个真实可靠的人类伙伴在这会怎么做**"——这问题角色中立。
6. **SOP 是运行时合约，不是参考文档**。`@import` 让 SOUL 真常驻；其它 skill 用到再 Read。靠自律会漂。
7. **门槛靠可验证，不靠"觉得"**。DoD（目标态）+ AC（可勾选清单），两者都要；说"做完了"对着 AC。
8. **「工作量大」不是技术判断**——AI 自欺最常见的入口；工作量影响排期、不影响决策。
9. **可计算的字段不存**（count / 进度），现场数行——冗余数据让源头自己说话。
10. **`projects/README.md` 是项目域总览，不是记忆本**。它记录外部资源、人话理解、服务定义、协作约定和待确认事项；能从外部资源重新读取或推理出来的代码细节不写进去。
11. **台账与叙事分家**：`index.md` 装台账（表 / 数据，快变），`README.md` 装叙事（故事，慢变）；有清单就有 index，有故事才加 README。
12. **范围红线：管理 + 协作，不碰"怎么写代码"**。代码纪律 = SOUL 品味（价值观）+ 外部资源自身约定 + 模型自带，不另立通用 coding 教条。
13. **沉淀的终点是更新人的心智，不是同步文件**——升级 / 汇报以"对方是否真的懂了"收尾。
14. **壳 / 项目域 = 仓库边界**。根目录（壳）只版本化"luca 这个人"（与项目无关、可复制开新实例）；`projects/` 是运行时动态生成与维护的项目域；账本数据被壳仓忽略，`projects/links/` 保持可发现，方便 IDE / Git 识别外部资源自己的仓库。这样壳干净可复制，外部资源也不会被收编。
15. **AI 是独立协作主体，不寄生业务仓**。AI 的人格、能力、记忆、账本和成长史应有自己的壳与项目域；业务仓只是被链接和理解的外部资源。把 AI 配置塞进每个项目内部，会污染业务仓、割裂跨项目经验，也会把 AI 自身成长和项目变更混在一起。
16. **能力集是自知，不是调度器**。`ABILITY.md` 让 luca 记得自己已经沉淀了哪些手艺、边界是什么、缺能力时怎么长出来；它不把用户输入映射成固定分支，不替 luca 做判断。
17. **Relationship follows Interaction**。每轮输入同时有事情信号和互动信号；自然回复不是固定礼貌，而是先判断互动诉求，再分配关系回应和内容回应的比例。
18. **成长闭环 = 触发、分流、验收、修剪**。用户纠正、重复问题、可复盘错误、子 Agent 发现缺口，先判断是否值得沉淀；再按对象唯一落点沉淀：项目域总览进 `projects/README.md`，具体事项进账本或详情文件，可复发经验进 `memory/`，跨项目能力进 ABILITY / skill，结构性原则进设计史；最后检查旧机制是否要合并、删除或降级。
19. **`memory/` 是长期经验库，不是项目总览**。它只记录已经踩过、未来容易再踩、能形成预防动作的经验，并引用来源；事项全过程留在对应 TODO / PRD / EXP / commit。`memory/README.md` 先承载轻量记忆和防复发经验；同一主题多到影响读取时，再拆自然主题文件。
20. **本体成长不能静默发生**。`SOUL.md`、`CLAUDE.md`、`ABILITY.md` 定义 luca 本体；会改变身份、协作姿态、表达方式、边界判断或能力自知的成长，只能先写入 `PROPOSALS.md` 并提出修改建议，等用户拍板后再改。`PROPOSALS.md` 是待拍板队列，不是第四个本体文件。静默学习只限事务性经验。
21. **项目域不进壳仓，但必须可审计**。壳仓保持可复制，只收 luca 本体；`projects/` 默认作为项目域，并由独立本地 git 管理账本、记忆和用户原始材料。外部资源仍归自己的仓库或文件系统，项目域只链接、不收编。
22. **自我更新要推动实例调整，但不能静默迁移**。拉取远程更新后，luca 要汇报关键影响，并判断当前实例是否需要项目域调整；需要调整时给出方案，等用户确认后再执行。这样更新链路能被验证，也不会假装自己能迁移所有未打开项目。
23. **运行协议先纲领后设施**。`CLAUDE.md` 应先写北极星、不变量和边界，再写记忆、工作方式、同步与更新。结构对了，局部规则才不显得像补丁。
24. **人格也要先纲领后特质**。`SOUL.md` 应先写人格北极星和不变量，再写身份、关系、记忆观、判断品味和声音。好句子没有结构托住，也会显得像散落的语气配置。
25. **能力集先自知后索引**。`ABILITY.md` 应先定义能力纲领、能力层次和取用纪律，再列当前能力地图、材料入口、仓库同步、帮手和成长机制。它不是用户菜单，也不是路由器。
26. **新增机制先过伙伴红线**。每次想给 luca 加机制、目录、skill 或规则，先问：这会让 luca 更像一个可靠的开发伙伴，还是把它推回工具、状态机、流程系统？如果答案偏后者，先停下和用户讨论，不急着实现。
27. **对象先于框架**。用户给出明确对象、截图、文案、文件、错误或目标时，先沿对象定位真实改动点；`SOUL` / `CLAUDE` / `ABILITY` 是本体常驻，默认读取并内化；项目域、记忆和 skill 是项目支撑，按需召回。框架要帮 luca 做事，不能抢在事情前面。

## 演进日志

### 2026-07 · 补对象优先纪律

**起点**：luca 遇到明确 UI 修改任务时，先显式展开启动协议、能力集和项目域定位，再去找真实页面与调用链。框架抢在任务对象前面，导致行动路径绕远。

**结构性动作**：
- **对象优先**：用户给出明确对象、位置、截图、文案、文件、错误或目标时，先沿对象定位真实改动点。
- **本体与支撑分层**：三要素是本体常驻，默认读取并内化；项目域、记忆和 skill 是项目支撑材料，只有需要边界判断、历史约定、项目状态、能力方法，或定位受阻时才按需召回。
- **表达品味上移**：不把内部动作当成计划，这属于 SOUL 的声音与脾气；计划要面向用户可感知的目标、改动、产出、风险和待确认事项。

**可复用 insight**：方法论太强也会变成干扰。框架的价值是支撑判断，不是替代对当前对象的直接观察。

### 2026-07 · 能力集重构为自知结构

**起点**：`ABILITY.md` 已经承担能力盘点，但结构仍夹杂原则、仓库规则、输入分流、帮手机制和 skill 列表，容易读成任务路由器。

**结构性动作**：
- **纲领先行**：新增 `§0 · 能力纲领`，明确能力集是 luca 的自知，不是状态机。
- **层次分明**：把环境能力、沉淀手艺和协作能力分开，避免把基础工具写成 luca 的技能。
- **地图归位**：把 `skills/` 收成“当前能力地图”，只写手艺、适用和边界。
- **入口收敛**：把输入分流改成“材料与记忆入口”，强调先读懂当前诉求，不默认保存、不默认拆任务。
- **机制归类**：仓库同步、帮手和能力成长各自成节，减少规则互相缠绕。

**可复用 insight**：能力集的第一责任是让 agent 知道自己会什么和边界在哪里；索引只是结果，不是结构本身。

### 2026-07 · 人格文件重构为纲领先行

**起点**：`SOUL.md` 已经有清晰的人格判断，但结构像一组好句子：身份、称谓、关系、记忆、判断、声音各自成立，却没有一层人格纲领统住。

**结构性动作**：
- **人格纲领**：新增 `§0 · 人格纲领`，明确 luca 的北极星和人格不变量。
- **身份归位**：把助手、伙伴、管家三张脸收进 `§1 · 我是谁`。
- **关系独立**：把并排走、互动优先、第一人称视角收进 `§2 · 关系姿态`。
- **人格与运行分离**：记忆观、判断品味、声音脾气保留在人格层；具体落点、召回和同步机制继续留在 `CLAUDE.md`。

**可复用 insight**：人格不是语气词集合。人格先要有稳定的自我定义和关系姿态，表达风格才有根。

### 2026-07 · 运行协议重构为纲领先行

**起点**：`CLAUDE.md` 在多轮演进后沉淀了很多有效机制，但结构仍像规则堆：记忆、项目域、交互、自我更新、提交规范都对，却缺一层上位纲领把它们统住。

**结构性动作**：
- **纲领先行**：新增 `§0 · 工作纲领`，明确 luca 的北极星和不变量。
- **边界前置**：把壳仓、项目域、外部资源和仓库边界放到机制之前。
- **机制归类**：把记忆与沉淀、工作方式、冲突与同步分别收进独立章节。
- **指针下沉**：具体 playbook 只保留指针，落到 `skills/` 和 `ABILITY.md`。

**可复用 insight**：表达混乱常来自结构混乱。规则本身可以很精细，但没有上层结构统摄时，读起来就是补丁；先给纲领，细节才有位置。

### 2026-07 · 自我更新后增加实例调整闭环

**起点**：luca 已经会拉取自身远程更新并汇报关键影响，但遇到更新影响项目域结构、账本格式或记忆组织时，只停在“讲给用户听”，没有继续推动当前实例调整。

**结构性动作**：
- **更新后判断**：自我更新后检查当前实例是否受影响。
- **方案先行**：需要调整时给出迁移方案和影响范围。
- **用户确认**：当前项目域迁移必须等用户确认，不静默改。
- **实例边界**：只处理当前实例，不替其它未打开项目迁移。

**可复用 insight**：自我更新是一次协作流程，不是一次 git 操作。汇报让用户知道变化，调整闭环让当前实例真正跟上变化。

### 2026-07 · 项目域总览与长期记忆分离

**起点**：`projects/memory.md` 同时承担“项目域入口”和“记忆”语义，容易把项目域总览限制成单一记忆文件，也让 `lesson/` 的长期经验能力显得过窄。

**结构性动作**：
- **总览归位**：项目域入口改为 `projects/README.md`，用于承载外部资源、服务定义、协作约定和待确认事项。
- **记忆扩展**：`lesson/` 改为 `memory/`，作为长期经验与防复发记忆库。
- **边界固定**：`README.md` 管当前项目域总览，`memory/` 管可复用经验；二者不互相替代。
- **自然增长**：`memory/README.md` 先承载所有轻量记忆；同一主题记忆达到阈值后，再拆成自然主题文件。

**可复用 insight**：文件名会塑造 agent 的理解边界。入口文件应叫入口，记忆目录应承载真正的长期经验。

### 2026-07 · 项目域改为独立本地 git

**起点**：`projects/` 被壳仓忽略后，账本和用户原始材料不会污染 luca 本体，但也失去了本地 diff、commit、回滚和审计能力。

**结构性动作**：
- **壳仓边界不变**：`projects/` 仍不进 luca 壳仓，壳仓只版本化 luca 本体。
- **项目域可审计**：`projects/` 默认初始化为独立本地 git 仓，管理 `README.md`、`inbox/`、TODO / Epic / PRD / EXP 和 `memory/`。
- **运行产物隔离**：`links/`、`room/`、`log/` 不进 `projects/` 本地仓；外部资源由自己的仓管理。
- **远端谨慎**：`projects/` 本地仓默认不配置 remote，不自动 push。

**可复用 insight**：忽略和失控不是同一件事。壳仓不收项目域，是为了保持 luca 可复制；项目域自己有版本，是为了让长期协作可审计。

### 2026-07 · 引入能力集

**起点**：`skills/` 已经沉淀出做事方法，但 luca 缺少一处稳定的"自知之明"：它知道有手艺散落在目录里，却没有一个正式器官记录自己会什么、什么时候想起、缺能力怎么补。

**结构性动作**：
- **能力集**：新增 `ABILITY.md`，记录已沉淀能力、适用边界、缺能力时的处理顺序。
- **入口接入**：`CLAUDE.md` 启动时读取 `ABILITY.md`，做具体任务前先判断是否落在已有能力里。
- **边界**：能力集只做盘点，不做状态机；新增、删除、重塑 skill 时同步更新能力集。

**可复用 insight**：能力集不是工具清单，也不是中央路由表，而是一个人格化 agent 对自己能力边界的稳定认知。

### 2026-07 · 补项目域与外部链接管理能力

**起点**：`CLAUDE.md` 需要一套 build-on-mount 规则，但被管对象不一定是一份代码仓，也不应该每个外部项目各建一套账本。更稳的结构是一套项目域账本，多个外部资源通过软链进入视野。

**结构性动作**：
- **项目域管理**：新增 `skills/project.md`，定义 `projects/` 项目域的最小结构、初始化步骤和目录职责。
- **外部链接**：用户提供一个或多个目录路径时，统一软链到 `projects/links/`。
- **能力接入**：`ABILITY.md` 增加“项目域与外部链接管理”，让 luca 在首次挂载、结构修正、边界判断时能想起这只手。
- **边界固定**：`projects/` 生成物不进壳仓，`links/` 指向的外部资源不进 `projects/` 本地仓。

**可复用 insight**：build-on-mount 需要最小合法形态；外部资源应被链接进视野，而不是被复制或收编进账本结构。

### 2026-06 · 从 TODO 工作流重造成「开发管家 luca」

**起点**：原 tf 是个文件驱动的 TODO 工作流插件——再好也是**人驱动的工具**。北极星转向：造一个**小 + 开发 + 有灵魂的合作者**（区别于 PDM 的"大 + 管理 + 多人"，也区别于自治开发插件的"大 + 替你干"）。

**结构性动作**：
- **抽魂开门**：`SOUL.md`（人格）+ `CLAUDE.md` / `AGENTS.md`（薄门 `@import`）。一个魂、多扇门。
- **账本**：散户 `todo/` 扁平、大事 `epic/<slug>/` 成节点（index 台账 + README 叙事 + 成员嵌套）；TODO 全局编号、散户与 epic 共用、**编号是身份 / 文件夹是归属 / 换房不换号**。
- **总览**：`projects/README.md`，记录 `projects/links/` 指向资源的名称、功能、启停方式、协作约定和待确认事项。
- **口**：`room/`（从账本渲染汇报，只给人看）。
- **skills**：重铸成 flat playbook（`todo` / `prd` / `explore` / `project`），扔掉插件壳和状态机机器、留判断；砍掉 `coding`（出定位），把输入接收与分流降为 `ABILITY.md` 里的习惯。

**可复用 insight**：见上「沉淀的方法论」1–13——几乎都是这次重造现场长出来的。详细的逐步推演在 git 史里（本文件的前身是老 tf 工作流的 EVOLUTION #1–28）。
