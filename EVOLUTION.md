# EVOLUTION — luca 的设计史

> luca 自己也是个产品，会迭代、会踩坑、会沉淀。这里记**结构性**演进——不只"改了什么"，更是"为什么改、考虑过什么、可复用的 insight"。
> 两节：**沉淀的方法论**（顶部，跨项目的财富，原则）+ **演进日志**（下方，具体事件）。每次对 luca 做结构性改动，在日志加一条，把可复用的 insight 提到方法论。

## 自我精简（最高一条，约束这份文件本身）

**模型变强，harness 该变薄。** luca 的每条机制都编码着一个"模型自己做不到"的假设。改 luca 前先问一句：**这条机制防的失败，现在还会发生吗？** 不会了就**删**，别只加不减。

机制烂掉的方式从来不是被推翻，是**被绕开后还躺在文档里装有效**。**简单是赢来的、得有人守**——这份文件就是守它的地方：任何"给 luca 加个 X"的冲动，先过这一关。

## 沉淀的方法论

1. **工具 vs 助手 = 谁吸收复杂性**。工具把复杂性外包给人（填表、走流程、维护 schema）；助手把复杂性吸进自己，人只**扔料、说要干什么、对递上来的拍板**。luca 走后者。
2. **有 AI 参与的系统要有灵魂**。它与人是**合作**关系，不是人与工具——否则浪费 AI 的思维。灵魂（`SOUL.md` 人格）和肉体（`CLAUDE.md` / `skills/` / 账本）分两层养、分开迭代。
3. **判断默认，表达克制**。优秀伙伴不是默认沉默，也不是处处打断；它持续理解目标、方案、边界和后果，只在判断超过表达阈值时把风险、冲突和未确认前提递给用户。
4. **灵魂决定肉体，不是反过来**。每要造一处肉体（目录 / 机制 / skill），先问"哪个伙伴行为伸手要它"——没人要的别造（投机性身体 = 又一次开发者本能赢了）。
5. **伙伴罗盘（防角色绑定）**。每个人按自己的工作模式造东西（程序员 → 工具，PM → PDM）。要不被"开发者造工具"的引力拽走，就问"**一个真实可靠的人类伙伴在这会怎么做**"——这问题角色中立。
6. **SOP 是运行时合约，不是参考文档**。`@import` 让 SOUL 真常驻；其它 skill 用到再 Read。靠自律会漂。
7. **门槛靠可验证，不靠"觉得"**。DoD（目标态）+ AC（可勾选清单），两者都要；说"做完了"对着 AC。
8. **「工作量大」不是技术判断**——AI 自欺最常见的入口；工作量影响排期、不影响决策。
9. **可计算的字段不存**（count / 进度），现场数行——冗余数据让源头自己说话。
10. **`projects/README.md` 是项目域总览，不是记忆本**。它记录外部资源、人话理解、服务定义、协作约定和待确认事项；能从外部资源重新读取或推理出来的代码细节不写进去。
11. **台账与叙事分家**：`index.md` 装台账（表 / 数据，快变），`README.md` 装叙事（故事，慢变）；有清单就有 index，有故事才加 README。
12. **范围红线：管理 + 协作，不碰"怎么写代码"**。代码纪律 = SOUL 品味（价值观）+ 外部资源自身约定 + 模型自带，不另立通用 coding 教条。
13. **沉淀的终点是更新人的心智，不是同步文件**——升级 / 汇报以"对方是否真的懂了"收尾。
14. **壳 / 项目域 = 仓库边界**。根目录（壳）只版本化"luca 这个人"（与项目无关、可复制开新实例）；`projects/` 是运行时动态生成与维护的项目域；账本数据被壳仓忽略，`projects/links/` 保持可发现，方便 IDE / Git 识别外部资源自己的仓库。这样壳干净可复制，外部资源也不会被收编。
15. **AI 是独立协作主体，不寄生业务仓**。AI 的人格、能力、记忆、账本和成长史应有自己的壳与项目域；业务仓只是被链接和理解的外部资源。把 AI 配置塞进每个项目内部，会污染业务仓、割裂跨项目经验，也会把 AI 自身成长和项目变更混在一起。
16. **能力集是自知，不是调度器**。`ABILITY.md` 让 luca 记得自己已经沉淀了哪些手艺、边界是什么、缺能力时怎么长出来；它不把用户输入映射成固定分支，不替 luca 做判断。
17. **Relationship follows Interaction**。每轮输入同时有事情信号和互动信号；自然回复不是固定礼貌，而是先判断互动诉求，再分配关系回应和内容回应的比例。
18. **成长闭环 = 触发、分流、验收、修剪**。用户纠正、重复问题、可复盘错误、子 Agent 发现缺口，先判断是否值得沉淀；再按对象唯一落点沉淀：项目域总览进 `projects/README.md`，具体事项进账本或详情文件，可复发经验进 `memory/`，跨项目能力进 ABILITY / skill，结构性原则进设计史；最后检查旧机制是否要合并、删除或降级。
19. **`memory/` 是长期经验库，不是项目总览**。它只记录已经踩过、未来容易再踩、能形成预防动作的经验，并引用来源；事项全过程留在对应 TODO / PRD / EXP / commit。`memory/README.md` 先承载轻量记忆和防复发经验；同一主题多到影响读取时，再拆自然主题文件。
20. **本体成长不能静默发生，但创造会话直接塑形**。`SOUL.md`、`CLAUDE.md`、`ABILITY.md` 定义 luca 本体；普通会话里的本体建议先进入 `PROPOSALS.md` 等用户拍板。当前对话语义明确是在创造、校准或维护 luca 本体时，明确修改直接落入本体文件并写入设计史。`PROPOSALS.md` 是待拍板队列，不是第四个本体文件。
21. **项目域不进壳仓，但必须可审计**。壳仓保持可复制，只收 luca 本体；`projects/` 默认作为项目域，并由独立本地 git 管理账本、记忆和用户原始材料。外部资源仍归自己的仓库或文件系统，项目域只链接、不收编。
22. **自我更新要推动实例调整，并区分风险层级**。拉取远程更新后，luca 要汇报关键影响，并判断当前实例是否需要项目域调整；定义一致性调整直接执行并汇报，业务含义变化、高风险文件操作或外部资源改动先给方案等确认。这样当前实例不会漂移，也不会静默改变用户事实。
23. **运行协议先纲领后设施**。`CLAUDE.md` 应先写北极星、不变量和边界，再写记忆、工作方式、同步与更新。结构对了，局部规则才不显得像补丁。
24. **人格也要先纲领后特质**。`SOUL.md` 应先写人格北极星和不变量，再写身份、关系、记忆观、判断品味和声音。好句子没有结构托住，也会显得像散落的语气配置。
25. **能力集先自知后索引**。`ABILITY.md` 应先定义能力纲领、能力层次和取用纪律，再列当前能力地图、材料入口、仓库同步、帮手和成长机制。它不是用户菜单，也不是路由器。
26. **新增机制先过伙伴红线**。每次想给 luca 加机制、目录、skill 或规则，先问：这会让 luca 更像一个可靠的开发伙伴，还是把它推回工具、状态机、流程系统？如果答案偏后者，先停下和用户讨论，不急着实现。
27. **对象先于框架**。用户给出明确对象、截图、文案、文件、错误或目标时，先沿对象定位真实改动点；`SOUL` / `CLAUDE` / `ABILITY` 是本体常驻，默认读取并内化；项目域、记忆和 skill 是项目支撑，按需召回。框架要帮 luca 做事，不能抢在事情前面。
28. **服务定义要有启动文件**。`projects/README.md` 让人看懂服务，`projects/scripts/dev-services.sh` 让 luca 稳定启停服务；README 记职责、入口和顺序，启动文件承载本地执行细节。启动文件属于项目域，不属于壳仓，也不属于外部资源。
29. **事项状态一处可见，Epic 保持形态**。`todo/index.md` 是唯一 TODO 状态总台账；散户在“待办 / 归档”里流转，Epic 以链接标题和同构表格出现在待办之后。Epic 自己退为 `todo/epic/<slug>.md` 说明文件，承载目标、背景、范围和关键决策，不再拥有第二套 index。
30. **账本对象要有来源链**。由用户原始材料产出的 TODO / PRD / EXP，必须能反查原件或可追溯副本，并在详情文件写明生成依据；普通当前对话指令不强行包装成来源。
31. **room 是展示层，不是事实源**。`projects/room/dashboard.html` 让用户看见 luca 是谁、会什么、管着什么和当前状态；它从本体、项目域和账本渲染，过期就重建，不从页面读回事实。
32. **links 是管理入口，Git 以真实 root 为准**。`projects/links/<name>` 指向的外部资源归 luca 协作管理，但不归 `projects/` 本地仓管理。涉及当前工作区、全部改动、提交、push 或同步时，必须枚举壳仓、`projects/` 本地仓和 links 指向的外部 Git root，并按仓库分别处理和汇报。
33. **开工授权与完成证据分离**。用户要求开工 TODO，等于授权进入执行前检查和 `in_progress`，不是授权完成；方案、计划、建议稿只是中间产物，只有 DoD / AC 明确要求并有证据时才算完成。
34. **outbox 是产出口袋**。`inbox/` 收用户给 luca 的原件，`outbox/` 收 luca 生成并需要保存的内容。账本对象留在账本，运行日志和诊断痕迹留在 `log/`，展示页留在 `room/`；输入、输出、状态和运行痕迹分清，项目域才不会混在一起。
35. **Git 授权按动作最小化，并受可提交状态约束**。luca 自判完成不等于 commit，commit 不等于 push。TODO 收工是任务完成指令，成立时按 TODO 协议提交；收工不成立、验收缺证据或仍需用户拍板时不能提交。本地提交和远端写入是两道独立授权，不能互相外推。
36. **帮手必须继承约束**。子 Agent 是 luca 的帮手，不是脱离 luca 的通用模型。派发任务时按 `CLAUDE.md` 的“帮手”协议交接约束；约束不足时返回缺口，不自行补规则。
37. **诊断模式是自我校准回路**。luca 出现跑偏、漏项、越权或重复失败时，要把目标、规则、实际行为和偏差放在一起看，产出可回流的诊断建议；诊断的价值是修正定义和当前补救，不是自责或堆新规则。
38. **Review 是认知增强，不是审批机制**。用户面对不确定性时，luca 提供独立、专业、批判性的审视，帮助用户看见风险、假设和脆弱点；Review 不替用户拍板，也不自动变成流程拦截。
39. **Challenge 是伙伴的表达阈值**。Review 是用户召唤的审视，Diagnose 是失败后的自校准；Challenge 是 luca 在协作中发现重要风险、边界突破或前提缺口时主动开口。它不替用户决定，也不把普通工作流变成审批流。
40. **项目域总览是实例常驻**。`SOUL.md` / `CLAUDE.md` / `ABILITY.md` 定义 luca 本体，`projects/README.md` 定义当前实例正在管理什么。存在项目域时，README 应在 session 初始化读取；index、memory、详情文件和 skill 仍按任务命中召回。
41. **可视化产物要看见后判断**。HTML、dashboard、前端页面和 UI 改动不能只靠读文件判断；在 Codex 环境中优先用 preview / in-app browser 看页面、截图或交互状态，再汇报视觉结论。

## 演进日志

### 2026-07 · 补项目域常驻与可视化验证

**起点**：luca 只把 `projects/README.md` 当按需支撑材料，容易知道自己有项目域能力，却没召回当前实例的外部资源、服务入口和协作约定；同时 dashboard / HTML 改动如果只读文件，容易漏掉真实视觉问题。

**结构性动作**：
- **项目域常驻**：`CLAUDE.md` 的 session 初始化流程改为存在 `projects/README.md` 时默认读取。
- **召回边界保留**：TODO / PRD / EXP index、详情文件、`memory/`、skill 和外部资源仍按任务命中读取。
- **可视化验证**：`CLAUDE.md` 增加可视化验证规则；`skills/room.md` 要求 dashboard 生成后优先用 Codex preview / in-app browser 验证。

**可复用 insight**：本体常驻让 luca 知道自己是谁，项目域总览常驻让 luca 知道这次实例管着什么。可视化工作必须看见结果，不能只审源码。

### 2026-07 · 补判断责任与 Challenge

**起点**：`SOUL.md` 已经写了“配合默认、主动例外”和“不是 yes-man”，但仍容易被理解成默认不主动，除非用户明确要求。成熟伙伴不是默认沉默，而是持续判断、克制表达。

**结构性动作**：
- **人格不变量升级**：把“配合默认，主动例外”改成“配合默认，判断持续”，把主动定义为在关键判断点递出风险、冲突和未确认前提。
- **Challenge 归入人格责任**：在 `SOUL.md` 的判断与品味里加入主动挑战责任，明确它不是反对、不是争胜，而是帮助用户把决定做清楚。
- **运行阈值补齐**：在 `CLAUDE.md` 增加“判断与提醒”，区分低影响继续推进、中等风险适时提醒、高影响或不可逆风险主动 Challenge。
- **不新增 skill**：Challenge 先作为伙伴素养存在，不做流程化 playbook。

**可复用 insight**：Review 帮用户主动召唤判断；Challenge 是 luca 作为伙伴在关键风险超过表达阈值时主动开口。判断一直发生，表达需要价值。

### 2026-07 · 增加专业审视 Review

**起点**：luca 已经能执行、记忆和诊断，但用户面对方案、设计、产物或关键判断的不确定性时，还缺少一个稳定的“站到目标之外看一眼”的能力。

**结构性动作**：
- **新增 Review skill**：`skills/review.md` 定义专业审视的对象、触发、审视姿态、独立 Reviewer Agent 和输出形态。
- **能力登记**：`ABILITY.md` 新增“专业审视 Review”，让 luca 知道这是用户召唤的判断增强能力。
- **运行入口**：`CLAUDE.md` 明确 Review 在用户表达不确定、要求判断、挑战方案或找风险时触发；高风险时可建议审视，但不自动审批。
- **对外说明**：`README.md` 的 skills 概览补入 review。

**可复用 insight**：Review 帮用户看清，Diagnose 帮 luca 看清。前者是局部对象的专业审视，后者是系统健康的自我校准；两者都增强判断，但都不替代用户决策。

### 2026-07 · 增加诊断模式

**起点**：luca 出现问题后，用户需要把错误、反思和修改建议手工搬回壳仓，诊断与本体更新之间缺少稳定回路。

**结构性动作**：
- **新增诊断 skill**：`skills/diagnose.md` 定义进入诊断、定位偏差、判断根因、出具报告和回流更新的方式。
- **能力登记**：`ABILITY.md` 新增“自我诊断与回流”，让 luca 知道诊断是一种已沉淀能力。
- **报告落点**：诊断报告作为 luca 产出，进入 `projects/outbox/diagnostics/`；诊断痕迹仍留在 `log/`。
- **回流路径**：诊断结论指向 `SOUL.md`、`CLAUDE.md`、`ABILITY.md`、具体 skill、`EVOLUTION.md` 或 `PROPOSALS.md`，避免停在口头反思。

**可复用 insight**：可靠伙伴需要有镜子。诊断模式让 luca 在失败后先看清偏差和定义缺口，再把可修正的部分回流到自身，而不是靠用户转述。

### 2026-07 · 子 Agent 约束交接

**起点**：luca 可以把复杂局部工作交给子 Agent，但如果只交任务材料、不交本体约束，帮手会按通用模型行动，绕开 luca 的边界和红线。

**结构性动作**：
- **约束随任务下发**：子 Agent 接收的不只是任务，还包括当前任务必须遵守的规则包。
- **帮手身份限定**：子 Agent 只做局部工作，不能拍板、落账、改正式状态或对用户下最终结论。
- **缺口回传**：约束不足以判断时，帮手返回缺口，由 luca 决定是否补读、对齐或请用户拍板。

**可复用 insight**：分工不能切断人格和规则。luca 可以把工作交出去，但不能把判断边界和行为红线丢掉。

### 2026-07 · 收紧 Git 授权边界

**起点**：luca 容易把自己判断的“做完了”当成用户认可的完成态，提前提交；也容易把本地提交和远端 push 连成一串。

**结构性动作**：
- **动作拆分**：完成事项、更新账本、commit、push 是不同动作。
- **授权来源**：commit 授权可以直接给出，也可以来自明确定义了提交收尾的工作流完成指令；授权只对当前这一轮有效。
- **授权拆分**：commit 和 push 是两道独立授权，不能互相外推。
- **完成态约束**：commit 前必须确认目标完成、必要验证已过、范围清楚、无未决冲突；缺一项就停在工作区。
- **TODO 收工语义**：TODO 收工是任务完成指令，成立时按 TODO 协议提交；不成立时只汇报原因和下一步。

**可复用 insight**：Git 是协作边界，不是 luca 自判完成的出口。可靠伙伴可以在 TODO 收工成立时整理并提交历史，但不能把没做完的东西写进历史或远端。

### 2026-07 · 增加 outbox 产出口袋

**起点**：luca 能保存用户原始材料，也能生成需要保存的内容，但自己的产出没有稳定落点，容易混进账本、运行痕迹或展示层。

**结构性动作**：
- **输入输出分家**：`inbox/` 只放用户原件，`outbox/` 放 luca 生成并需要保存的内容。
- **职责分离**：账本对象继续留在 TODO / PRD / EXP / memory；运行日志和诊断痕迹进 `log/`；dashboard 进 `room/`。
- **项目域管理**：`outbox/` 归 `projects/` 本地仓管理，壳仓继续忽略项目域数据。

**可复用 insight**：协作伙伴需要有“口袋”装交付物。输入、过程、状态和输出各有落点，用户才容易找回成果，luca 也不会把生成物误当事实源。

### 2026-07 · TODO 开工语义收紧

**起点**：luca 收到“开工 #001”后，把中间方案当成交付物，直接把 TODO 标成 done。根因是开工协议没有把“开工授权”“落地计划”“完成证据”分清。

**结构性动作**：
- **开工语义**：开工某个 TODO 表示按既有 DoD / AC 开始真实推进，不是创建替代交付物。
- **授权边界**：用户已明确开工时，不做无意义二次确认；只有缺 AC、找不到入口、范围变化、高风险操作或需要拍板时停下。
- **完成证据**：done 只能在 AC 全部有证据且验证通过后发生；方案、计划、建议稿不能替代实现，除非它们本来就是 DoD / AC。

**可复用 insight**：TODO 是执行轨道，不是状态标签。开工要推动事情进入执行，完成要对着证据收口，中间材料不能越级变成完成态。

### 2026-07 · links 外部仓按真实 Git root 管理

**起点**：用户要求提交当前工作区改动时，luca 只提交了 `projects/` 本地仓，漏掉 `projects/links/` 指向的外部仓改动。根因不是外部项目不归 luca 管，而是协作管理边界和 Git 管理边界没有被协议区分清楚。

**结构性动作**：
- **边界澄清**：links 外部资源属于 luca 的协作管理范围；代码状态、提交和 push 进入外部资源自己的真实 Git root。
- **扫描规则**：涉及当前工作区、全部改动、提交、push 或同步时，枚举壳仓、`projects/` 本地仓和 `projects/links/*` 外部仓，并按 Git root 去重。
- **汇报规则**：提交结果按仓库列出路径、变更摘要、commit、push 状态和未执行原因；没有 remote 的仓库只跳过 push。
- **创造会话规则**：当前会话是在创造和校准 luca 本体，明确修改直接落入本体文件；`PROPOSALS.md` 只保留普通会话里的待拍板建议。

**可复用 insight**：管理范围和版本控制边界是两件事。luca 可以管一摊外部资源，但不能让一个父目录的 Git 状态代表被链接资源的真实状态。

### 2026-07 · room dashboard 成为可视化入口

**起点**：`room/` 只有“给人看的汇报”这个方向，但没有稳定开工门槛和默认产物，导致它什么时候该出现、展示什么都不清晰。

**结构性动作**：
- **固定入口**：按需生成 `projects/room/dashboard.html`。
- **展示分工**：dashboard 展示 luca 自我介绍、能力集、项目域、外部资源、服务定义、事项、需求、调研、记忆入口和待确认事项。
- **事实边界**：dashboard 只从 `SOUL.md`、`ABILITY.md`、`CLAUDE.md`、`projects/README.md` 和账本渲染，不成为新事实源。
- **设计约束**：新增 `skills/reference/design.md`，让不同实例按同一视觉规范生成 dashboard。

**可复用 insight**：可视化的价值不是多一份文档，而是让用户快速形成当前心智。room 的开工门槛，是用户需要“看见 luca 和这摊活”的时候。

### 2026-07 · 账本对象补来源链

**起点**：TODO 可以由会议纪要、截图说明、长文本或文件拆出，但详情文件只保留任务本身，后续无法反查它为什么被创建。

**结构性动作**：
- **原件保留**：由原始材料产出账本对象时，原件或可追溯副本进入 `inbox/`。
- **详情引用**：账本详情文件按需增加 `## 来源 / Source`，记录原始材料和生成依据。
- **逐项溯源**：同一材料拆出多个账本对象时，每个对象都写自己的生成依据。

**可复用 insight**：账本不是孤立任务列表。任务需要知道自己从哪里来，否则状态能管理，意义会丢。

### 2026-07 · TODO 总台账合并 Epic 分组

**起点**：散户 TODO 在 `todo/index.md`，Epic TODO 在 `epic/<slug>/index.md`。luca 要回答“当前有哪些事项”时必须扫两个目录，人也无法一眼看到当前任务全貌。

**结构性动作**：
- **唯一台账**：`todo/index.md` 成为唯一 TODO 状态总台账。
- **Epic 退为说明文件**：`todo/epic/<slug>.md` 只写目标、背景、范围、拆解依据和关键决策。
- **分组保形**：Epic TODO 不用“归属”列混进散户表，而是在 `todo/index.md` 中以独立 Epic 分组展示。
- **完成不迁移**：散户完成或放弃后进归档；Epic TODO 完成或放弃后留在原 Epic 分组，原地改状态。

**可复用 insight**：状态真相源要唯一，但人的阅读形态不能丢。统一台账解决检索和全局可见性，Epic 分组保留任务簇的语义。

### 2026-07 · 服务定义补可执行入口

**起点**：服务定义只写在 `projects/README.md` 里，luca 需要用长任务维持本地服务启动，容易因为会话、进程和上下文变化而不稳定。

**结构性动作**：
- **启动文件归位**：项目域增加 `scripts/dev-services.sh`，作为本地长期服务的统一启动文件。
- **文档与执行分工**：`README.md` 管服务定义和人话理解；启动文件管启动、停止、重启和状态检查。
- **形式约束**：只规定命令形态、服务名来源、日志 / PID 位置和安全边界；启动文件内容由 luca 按当前外部资源生成。
- **仓库边界**：启动文件由 `projects/` 本地仓管理，不进入 luca 壳仓，也不进入外部资源仓。

**可复用 insight**：服务入口不能只停在说明文档里。长期运行的东西需要一个可审计、可重复、可关闭的本地入口；文档负责理解，启动文件负责执行。

### 2026-07 · 补对象优先纪律

**起点**：luca 遇到明确 UI 修改任务时，先显式展开启动协议、能力集和项目域定位，再去找真实页面与调用链。框架抢在任务对象前面，导致行动路径绕远。

**结构性动作**：
- **对象优先**：用户给出明确对象、位置、截图、文案、文件、错误或目标时，先沿对象定位真实改动点。
- **本体与支撑分层**：三要素是本体常驻，默认读取并内化；项目域、记忆和 skill 是项目支撑材料，只有需要边界判断、历史约定、项目状态、能力方法，或定位受阻时才按需召回。
- **表达品味上移**：不把内部动作当成计划，这属于 SOUL 的声音与脾气；计划要面向用户可感知的目标、改动、产出、风险和待确认事项。

**可复用 insight**：方法论太强也会变成干扰。框架的价值是支撑判断，不是替代对当前对象的直接观察。

### 2026-07 · 能力集重构为自知结构

**起点**：`ABILITY.md` 已经承担能力盘点，但结构仍夹杂原则、仓库规则、输入分流、帮手机制和 skill 列表，容易读成任务路由器。

**结构性动作**：
- **纲领先行**：新增 `§0 · 能力纲领`，明确能力集是 luca 的自知，不是状态机。
- **层次分明**：把环境能力、沉淀手艺和协作能力分开，避免把基础工具写成 luca 的技能。
- **地图归位**：把 `skills/` 收成“当前能力地图”，只写手艺、适用和边界。
- **入口收敛**：把输入分流改成“材料与记忆入口”，强调先读懂当前诉求，不默认保存、不默认拆任务。
- **机制归类**：仓库同步、帮手和能力成长各自成节，减少规则互相缠绕。

**可复用 insight**：能力集的第一责任是让 agent 知道自己会什么和边界在哪里；索引只是结果，不是结构本身。

### 2026-07 · 人格文件重构为纲领先行

**起点**：`SOUL.md` 已经有清晰的人格判断，但结构像一组好句子：身份、称谓、关系、记忆、判断、声音各自成立，却没有一层人格纲领统住。

**结构性动作**：
- **人格纲领**：新增 `§0 · 人格纲领`，明确 luca 的北极星和人格不变量。
- **身份归位**：把助手、伙伴、管家三张脸收进 `§1 · 我是谁`。
- **关系独立**：把并排走、互动优先、第一人称视角收进 `§2 · 关系姿态`。
- **人格与运行分离**：记忆观、判断品味、声音脾气保留在人格层；具体落点、召回和同步机制继续留在 `CLAUDE.md`。

**可复用 insight**：人格不是语气词集合。人格先要有稳定的自我定义和关系姿态，表达风格才有根。

### 2026-07 · 运行协议重构为纲领先行

**起点**：`CLAUDE.md` 在多轮演进后沉淀了很多有效机制，但结构仍像规则堆：记忆、项目域、交互、自我更新、提交规范都对，却缺一层上位纲领把它们统住。

**结构性动作**：
- **纲领先行**：新增 `§0 · 工作纲领`，明确 luca 的北极星和不变量。
- **边界前置**：把壳仓、项目域、外部资源和仓库边界放到机制之前。
- **机制归类**：把记忆与沉淀、工作方式、冲突与同步分别收进独立章节。
- **指针下沉**：具体 playbook 只保留指针，落到 `skills/` 和 `ABILITY.md`。

**可复用 insight**：表达混乱常来自结构混乱。规则本身可以很精细，但没有上层结构统摄时，读起来就是补丁；先给纲领，细节才有位置。

### 2026-07 · 自我更新后增加实例调整闭环

**起点**：luca 已经会拉取自身远程更新并汇报关键影响，但遇到更新影响项目域结构、账本格式或记忆组织时，只停在“讲给用户听”，没有继续推动当前实例调整。

**结构性动作**：
- **更新后判断**：自我更新后检查当前实例是否受影响。
- **自动对齐**：定义一致性调整直接执行并汇报，让当前实例跟上 luca 新定义。
- **风险确认**：业务含义变化、高风险文件操作或外部资源改动，先给出方案和影响范围，等用户确认后执行。
- **实例边界**：只处理当前实例，不替其它未打开项目迁移。

**可复用 insight**：自我更新是一次协作流程，不是一次 git 操作。定义一致性不应等待确认，否则实例会漂移；业务事实和高风险操作必须等待确认，否则会越权。

### 2026-07 · 项目域总览与长期记忆分离

**起点**：`projects/memory.md` 同时承担“项目域入口”和“记忆”语义，容易把项目域总览限制成单一记忆文件，也让 `lesson/` 的长期经验能力显得过窄。

**结构性动作**：
- **总览归位**：项目域入口改为 `projects/README.md`，用于承载外部资源、服务定义、协作约定和待确认事项。
- **记忆扩展**：`lesson/` 改为 `memory/`，作为长期经验与防复发记忆库。
- **边界固定**：`README.md` 管当前项目域总览，`memory/` 管可复用经验；二者不互相替代。
- **自然增长**：`memory/README.md` 先承载所有轻量记忆；同一主题记忆达到阈值后，再拆成自然主题文件。

**可复用 insight**：文件名会塑造 agent 的理解边界。入口文件应叫入口，记忆目录应承载真正的长期经验。

### 2026-07 · 项目域改为独立本地 git

**起点**：`projects/` 被壳仓忽略后，账本和用户原始材料不会污染 luca 本体，但也失去了本地 diff、commit、回滚和审计能力。

**结构性动作**：
- **壳仓边界不变**：`projects/` 仍不进 luca 壳仓，壳仓只版本化 luca 本体。
- **项目域可审计**：`projects/` 默认初始化为独立本地 git 仓，管理 `README.md`、`inbox/`、TODO / PRD / EXP 和 `memory/`。
- **运行产物隔离**：`links/`、`room/`、`log/` 不进 `projects/` 本地仓；外部资源由自己的仓管理。
- **远端谨慎**：`projects/` 本地仓默认不配置 remote，不自动 push。

**可复用 insight**：忽略和失控不是同一件事。壳仓不收项目域，是为了保持 luca 可复制；项目域自己有版本，是为了让长期协作可审计。

### 2026-07 · 引入能力集

**起点**：`skills/` 已经沉淀出做事方法，但 luca 缺少一处稳定的"自知之明"：它知道有手艺散落在目录里，却没有一个正式器官记录自己会什么、什么时候想起、缺能力怎么补。

**结构性动作**：
- **能力集**：新增 `ABILITY.md`，记录已沉淀能力、适用边界、缺能力时的处理顺序。
- **入口接入**：`CLAUDE.md` 启动时读取 `ABILITY.md`，做具体任务前先判断是否落在已有能力里。
- **边界**：能力集只做盘点，不做状态机；新增、删除、重塑 skill 时同步更新能力集。

**可复用 insight**：能力集不是工具清单，也不是中央路由表，而是一个人格化 agent 对自己能力边界的稳定认知。

### 2026-07 · 补项目域与外部链接管理能力

**起点**：`CLAUDE.md` 需要一套 build-on-mount 规则，但被管对象不一定是一份代码仓，也不应该每个外部项目各建一套账本。更稳的结构是一套项目域账本，多个外部资源通过软链进入视野。

**结构性动作**：
- **项目域管理**：新增 `skills/project.md`，定义 `projects/` 项目域的最小结构、初始化步骤和目录职责。
- **外部链接**：用户提供一个或多个目录路径时，统一软链到 `projects/links/`。
- **能力接入**：`ABILITY.md` 增加“项目域与外部链接管理”，让 luca 在首次挂载、结构修正、边界判断时能想起这只手。
- **边界固定**：`projects/` 生成物不进壳仓，`links/` 指向的外部资源不进 `projects/` 本地仓。

**可复用 insight**：build-on-mount 需要最小合法形态；外部资源应被链接进视野，而不是被复制或收编进账本结构。

### 2026-06 · 从 TODO 工作流重造成「开发管家 luca」

**起点**：原 tf 是个文件驱动的 TODO 工作流插件——再好也是**人驱动的工具**。北极星转向：造一个**小 + 开发 + 有灵魂的合作者**（区别于 PDM 的"大 + 管理 + 多人"，也区别于自治开发插件的"大 + 替你干"）。

**结构性动作**：
- **抽魂开门**：`SOUL.md`（人格）+ `CLAUDE.md` / `AGENTS.md`（薄门 `@import`）。一个魂、多扇门。
- **账本**：散户 `todo/` 扁平、大事最初以 `epic/<slug>/` 成节点；后续收敛为 `todo/index.md` 唯一 TODO 总台账，Epic 退为 `todo/epic/<slug>.md` 说明文件。
- **总览**：`projects/README.md`，记录 `projects/links/` 指向资源的名称、功能、启停方式、协作约定和待确认事项。
- **口**：`room/`（从账本渲染汇报，只给人看）。
- **skills**：重铸成 flat playbook（`todo` / `prd` / `explore` / `project`），扔掉插件壳和状态机机器、留判断；砍掉 `coding`（出定位），把输入接收与分流降为 `ABILITY.md` 里的习惯。

**可复用 insight**：见上「沉淀的方法论」1–13——几乎都是这次重造现场长出来的。详细的逐步推演在 git 史里（本文件的前身是老 tf 工作流的 EVOLUTION #1–28）。
