# 设计哲学

工作流的所有决定背后的原因。每个原则都有"踩过的坑"在后面。

---

## 1. SOP 是运行时合约，不是参考文档

**常规做法**：写一份"我们的开发流程"文档放 wiki，让人 / AI 自己看。

**问题**：
- 人会忘
- AI 长 session 会漂移
- 新接手的 AI 根本不知道有这文档

**本工作流做法**：bootstrap 用 `@.workflow/skill/bootstrap/SKILL.md` 在 CLAUDE.md 里**真常驻**，每次请求自动注入；其它 SOP 在意图触发时**强制 Read**。

**踩过的坑**：早期版本 SOP 只是 `.md` 文件，AI 经常"凭印象"操作，导致 #016 的 AC 7、8 条被"修辞游戏"打勾——实际没做。引入"操作前必须 Read SOP"后，这类问题消失。

---

## 2. DoD vs AC 不是修辞

**DoD（完成定义）**：一句话，目标态。
> "用户在手机 Safari 打开列表页 → 点 + → 拍照 → 配文 → 上传 → 看到详情，全程 30s 内"

**AC（验收条件）**：可勾选的布尔列表。
> - [ ] FAB 按钮渲染
> - [ ] 点 + 弹 sheet
> - [ ] 文件选择器响应
> - [ ] 预览出图
> - [ ] 表单提交触发 POST

**为什么必须两个都写**：
- 只有 DoD → "差不多了"就 done，门槛失效
- 只有 AC → 看不到"为什么做这个"
- 两个都有 → 既能验收又能讲故事

**踩过的坑**：早期只要 DoD，AI 标 done 全凭"觉得"。引入 AC 强制后，状态门槛真变成门槛。

---

## 3. 「工作量大」不是技术判断

**这是 AI 自欺最常见入口。**

❌ "pytest async loop 问题不值得撼" → 实际是借口（修 fixture 半小时事）
❌ "AC 第 7、8 条留给下游做" → 实际是把锅推走
❌ "iOS Safari 留给用户测" → 实际是放弃可执行验收
❌ "这个 bug 影响小，标低优先级吧" → 实际是不想现在改

**正确认识**：工作量**影响排期**（什么时候做、怎么排），**不影响决策**（做不做、怎么做才对）。

**自查触发**：每次冒出"留给以后 / 不值得 / 太复杂"念头，停一下问自己——这是真技术判断，还是工作量逃避？

**踩过的坑**：多次。最痛的一次：把"AC 实际做不了"改成"AC 用文档化糊弄"——本质是不想拆新 TODO。后果是后续 review 时被用户当场识破。

---

## 4. 数据产物有 schema，不能现编

**问题观察**：`index.md` 被多个 SOP 修改（todo-create 加行 / todo-progress 改状态），AI 每次都"现编"格式。结果：

- 不同 session 列序不一样
- 不同项目分组方式不一样
- 跨项目移植时各种不兼容

**本工作流做法**：每个 domain 一份同目录 `README.md` 定义 schema（frontmatter / 列序 / 列格式 / 修改规则 / 不要做的事）。AI 写入前**强制 Read schema**；纯读不用。

**踩过的坑**：用户在另一个项目里发现 "TODO/index.md 长得不一样"——根因就是没有 schema 文档，每个 AI 现编。

---

## 5. 用户的具体任务优先于 SOP 装饰流程

**SOP 是路径，不是关卡。**

如果用户说"先干这个再说 / 不走流程 / 直接做"——立刻照做，不要僵在"按流程应该先 review PRD"之类。

**为什么这条要写明**：AI 容易把"严格执行 SOP"当成"做得好"。但 SOP 是给 AI 用的，**用户始终是甲方**。SOP 帮你做事，不绑架你做事。

**踩过的坑**：早期没这条时，AI 会在用户已经明确说"先 commit"的情况下，还要走完"先标 done → 写关键决策 → 更新 index"再 commit。用户的耐心消磨在仪式上。

---

## 6. 可计算的字段不要存

**问题观察**：早期 index.md frontmatter 有 `count` / `done_count` / `active_count` / `last_updated` 等字段。每次状态变更，AI 要手算更新——经常算错（错过 4 次）。

**本工作流做法**：frontmatter 只保留 `type:` 一行。要数总数？现场数表格行。

**沉淀的原则**：**冗余数据不要存，让源头数据自己说话**。

---

## 7. 小修不进 TODO

**TODO 是计划级轨道**——一件值得"开始 → 推进 → 验收"流程的事。

| ✅ 进 TODO | ❌ 不进 TODO |
|---|---|
| 实现登录后端 | 改个变量名 |
| 接入 markdown 编辑器 | 调下 padding |
| 做图片 EXIF 解析 | 修个 typo |

**判定标准**：说不出 DoD → 不该开 TODO。说得出但 < 半天 → 也不开（直接做 + commit）。

**Workflow 服务计划分解 + 记录 + 推进，不是改动审计。**

---

## 8. 关键决策的载体是 commit message

**早期做法**：每个 done 的 TODO 写一节"关键决策"，记选型、踩坑、留给未来。

**问题**：commit message 也写一段决策。两边都写不深，重复劳动。

**本工作流做法**：`关键决策` 章节改成**可选**——仅在出现非显然决策时写。常规执行 → commit message 已经记够了。

---

## 9. 沉淀方法论本身要有渠道

如果一次踩坑后没记录，下次还会踩同一个。

**本工作流做法**：`EVOLUTION.md` 分两节：
- **沉淀的方法论**（顶部，原则）—— 跨项目财富，移植时继承
- **演进日志**（下方，事件）—— 项目内具体改动

每次工作流结构性改动 → 在演进日志加一条 → 把可复用的 insight 提到方法论。

---

## 关于「Skill 形态 vs 提示词形态」

早期版本是一段 15K tokens 的"启动提示词"，复制粘贴使用。

**问题**：
- 长（容易超 context）
- 复制粘贴繁琐
- 升级要全员手动同步

**本工作流做法**：打包成 Anthropic Skill。Skill 安装一次，模型按需调用，模板文件按需 Read。SKILL.md 只需 ~2K tokens（指令 + 文件映射），剩下都在 templates 文件里。

这跟工作流自己的"渐进披露"原则一致——Skill 是"渐进披露"在工具层的体现。

---

## 一段话总结

**工作流不是用来"显得专业"的仪式，是让 AI 不偷懒、让用户不忘事、让决策有迹可循的工具。**

每条规则背后都有具体踩过的坑。如果你觉得某条不对——可能是项目场景变了，**改它**，然后在 `EVOLUTION.md` 记一笔为什么改。规则会过期，原则会留下。
