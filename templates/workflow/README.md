# .workflow/ — 工作流环境

文件驱动的产品 / 工程管理。**新 session 入口**：[`skill/bootstrap/SKILL.md`](skill/bootstrap/SKILL.md)。

## 这是什么

一套**个人 / 小团队 + AI 协作**的工作范式：念头进 **EXP**，成熟后视体量长成 **PRD**（重）或直接成 **TODO**（轻）；**TODO 是一等行动单元**（自带总纲 + 决策，来源不止 PRD）；**SKILL 是每个环节的运行时合约**——既驱动流转，又卡质量门槛，让模型走标准流程且不偷懒漂移。

三个对象 + 一个驱动：

| | 是什么 | 形态 |
|---|---|---|
| **EXP**（预研 / 灵感池） | 还没想清的念头。`seed → exploring → promoted / dropped` | 业务 / 研究项目 |
| **PRD**（产品需求） | 业务现状 + 设计 / 技术设计 + 决策。你写、AI review、拆 TODO。**可选**——工具/库项目没有 | 业务 / 产品项目 |
| **TODO**（行动单元） | 一等公民，自带总纲（背景）+ 关键决策/方案。来源四种：PRD 拆解 / EXP promote / 临时需求 / 执行中衍生 | 所有项目 |
| **SKILL**（运行时合约） | 驱动上述流转 **+ 卡门槛**（AC 不齐不开工、验证不过不 done、红线防偷懒）。这套护栏是它区别于普通待办清单的灵魂 | 所有项目 |

> 典型流：念头 → EXP → 成熟 → PRD（大）/ TODO（小）→ 模型按 SKILL 推执行。
> 但**不是瀑布**：小事、临时需求、执行中发现的，直接建 TODO，不经 EXP / PRD。
> **Epic** 是把"同一件事的多个 TODO"捆起来的视图，不是独立对象。

## Skill 文件结构约定

每个 Skill 是一个独立目录：

```
skill/<skill-name>/
├── SKILL.md        ← 主文件（必有）。Bootstrap 加载它。
├── TEMPLATE.md     ← 可选：使用时复制的模板
├── EXAMPLES.md     ← 可选：示例
└── ...             ← 其它资源（脚本、参考、checklist）
```

匹配 Anthropic Skills / Cursor Rules 等业界 Skill 体系约定，便于未来复用。

## 设计原则

1. **文件驱动**：所有状态在文件里，不在脑子里
2. **渐进披露**：AI 先读 index 再读详情，省 token
3. **SOP 作为运行时合约**：操作前必须 Read 对应 SKILL.md，不能凭记忆（详见 bootstrap）
4. **PM ↔ 工程师**：一个人戴两顶帽子，PRD 与 PRD-R 强迫切换视角
5. **永不复用编号**：done / abandoned 的 ID 也保留，方便追溯
6. **永不静默改状态**：所有变更同步 `last_updated`
7. **工作流本身要演进**：每次对工作流的结构性改动后，在 [`EVOLUTION.md`](EVOLUTION.md) 加一条——不只是 changelog，更是方法论沉淀

## Skill 加载模型（重要）

这些 SKILL.md **不是** Anthropic Skills 系统的真 Skill（不会出现在 Claude Code 的 `available skills` 里）。它们是结构化成 Skill 形态的 SOP。加载分三层：

```
Layer 1 — 真常驻（每次请求自动注入）
└── CLAUDE.md
    └── @.workflow/skill/bootstrap/SKILL.md   ← Claude Code 的 @ 导入语法，bootstrap 内容被内联

Layer 2 — 按意图 Read（bootstrap 路由表指挥）
├── todo-create/SKILL.md
├── todo-progress/SKILL.md
├── prd-review/SKILL.md
└── explore/SKILL.md

Layer 3 — Skill 内部资源
└── todo-create/TEMPLATE.md  等
```

**为什么只 `@` 导入 bootstrap，不 `@` 其它 Skill**：
- bootstrap 是路由器，必须时刻在
- 其它 Skill 是功能模块，按需读取，避免每次请求都把全部 SOP 塞进上下文
- 这就是"渐进披露"原则的落地

**对其它 AI 工具的兼容**：Codex 等不识别 `@` 语法的工具，会把 `@.workflow/...` 当字面文本看到——通常会被识别为"该读这个文件"的暗示，最坏情况手动 Read 即可，功能不破。

## 升级：把工作流同步到上游最新

> 本工作流的真相源是**本机的 `todo-workflow-skill` 工作区**（SOP / schema 在那里演进，**始终最新**）。
> 约定它装在 `~/.claude/skills/todo-workflow`（维护者把这个路径软链到自己的开发工作区，所以它永远指向最新）。
> 每个项目装的是某一刻的快照，会过时。**升级 = 拿本项目和这个本地目录对比，不走远程 git**（本地比对零成本）。
> 本节是给**任何项目里的 AI**看的升级 SOP——用户说"更新 / 升级 / 同步工作流"、"update workflow"时，按本节执行。

### 两种触发（缺一不可）

| 触发 | 何时 | 行为 |
|---|---|---|
| **主动检测** | **仅新会话启动**时（bootstrap Step 1 顺带比对版本） | 落后才在首条回应顶部冒一行升级提示；不自动升 |
| **手动触发** | **任意时刻，不限会话开头** | 用户说"更新工作流 / 检查工作流更新" → 立即按本节执行 |

> **为什么两个都要**：主动检测基于会话启动——但用户常在**一个长会话里连续干活**，中途没有"新会话"，主动提示不会再冒。这时靠手动："更新工作流"随时能发，**长任务进行中也能升**。
> **任务中升级是安全的**：升级只动 `.workflow/` 里的 SOP/schema + index 结构，**不碰你正在改的业务代码**。唯一要点：升级后**必须重读被覆盖的 SOP**（含 bootstrap）刷新认知——这是 step 4「重新载入认知」，**别跳**（内存里是旧版，启动协议又不会中途重注入）。升级完回到原任务。

### 心智模型：代码层 vs 数据层（**升级安全的命根子**）

`.workflow/` 里两类文件混在一起，升级时必须分清：

| 层 | 是什么 | 哪些文件 | 升级动作 |
|---|---|---|---|
| **代码层**（上游所有，项目无关） | SOP + schema | 所有 `README.md`（含本文件）+ 整个 `skill/` 树 | **整体覆盖** |
| **数据层**（项目独有） | 你的实际内容 | 各域 `index.md`、`TODO_*/PRD_*/EXP_*` 详情、`projects.md` | **原样保留，绝不覆盖** |
| 混合 | 方法论(上游) + 演进日志(项目) | `EVOLUTION.md` | 刷新上游「沉淀的方法论」列表，保留项目「演进日志」 |
| 混合 | 注入 + 项目说明 | `CLAUDE.md` / `AGENTS.md` | 只更新工作流段，不碰项目内容 |

> 一句话记牢：**`README.md` 和 `skill/` 是代码，`index.md` 和带编号的详情文件是数据。代码覆盖，数据不动。**

### 升级步骤

> **"只检查"走到 step 2 停**：报出落后几版 + 差了哪些条目，问用户"要现在升吗"，得到同意再往 step 3 走。
> **"更新/升级"一路走完** step 1→9。

1. **定位本地真相源**（不 clone、不 pull）：
   - 真相源模板根 = `~/.claude/skills/todo-workflow/templates/workflow/`（约定路径，软链到维护者工作区，始终最新）。
   - 该路径不存在 / 找不到 → 让用户指出本机 `todo-workflow-skill` 工作区路径，用它的 `templates/workflow/`。
   - 全程只读本地文件系统，无网络往返。

2. **算版本差**：对比真相源与本项目 `EVOLUTION.md` 的「## 沉淀的方法论」**最大编号**。
   - 编号相同 → 多半只是文字微调，覆盖代码层即可，**无需数据迁移**。
   - 真相源更高 → 差出来的那几条**就是这次要落地的结构性变更 + 数据迁移指南**，逐条读。

3. **覆盖代码层**（从真相源 `templates/workflow/` 复制，仅限）：
   - 所有 `*/README.md`（含根 `README.md`）
   - 整个 `skill/` 树（全部 `SKILL.md` / `TEMPLATE.md`）
   - ⚠️ **绝不复制真相源的 `index.md`**（那是空模板，会清空你的数据）、`projects.md`、任何 `TODO_*/PRD_*/EXP_*`。

4. **重新载入认知**（**任务中升级的命门——别跳**）：覆盖完，把接下来要依赖的、刚被覆盖的文件**重新 Read 进上下文**：
   - **必读新 bootstrap**（`skill/bootstrap/SKILL.md`）——会话开头注入的那份现在是旧的，路由表 / 红线可能已变。
   - **要迁数据 → 先读新 schema**（`todo/README.md` 等），否则 step 5 会按旧结构迁错。
   - 本 session 已加载、接下来还要用的其它 SOP，一并重读。
   - ⚠️ **为什么这步靠升级 SOP、而不是启动协议**：bootstrap 是**会话开头注入的**，任务中途升级**不会让它自动重注入**——指望"启动协议提醒"在长任务里根本不会再触发。**只有把重读写成升级 SOP 的强制步骤，AI 的认知才跟得上刚落地的新版。**

5. **按 EVOLUTION 新增条目迁移数据**：每条结构性变更对照本项目 `index.md` 落地（**只改结构、不改语义**）。
   - 例（#22）："Epic 加 `完成` 列 + 完结整块迁到 `## 已完结专项`" → 给每张 Epic 表补 `完成` 列、把全绿 Epic 块下沉、补 `## 已完结专项` 段。
   - 加列 / 加分区 / 重排可以；删条目 / 改 TODO 内容**不行**。

6. **刷新 EVOLUTION.md**：真相源新增的方法论条目追加进本项目「## 沉淀的方法论」；本项目自己的「## 演进日志」时间线**原样保留**。

7. **刷新 CLAUDE.md 工作流段**：仅当 `@.workflow/skill/bootstrap` 注入块或紧邻的 Skill 清单有变化时更新；项目简介 / 技术栈 / 编码约定**不动**。

8. **自检**（机械层，先确保没改坏）：
   - `grep -rn "{{" .workflow/` 无残留占位符
   - 各 `index.md` 表头列序与对应 `README.md` 新 schema 一致

9. **心智简报（本次升级真正的产出——更新的是人的心智，不是文件）**：

   > "覆盖了 N 个文件、从 #N 升到 #M" 这种报告**没有价值**——用户读完，明天用工作流的方式不会有任何不同。
   > 升级的终点是：**用户知道从现在起哪里不一样了，该怎么改习惯。** 文件同步只是手段。

   对**每一条**版本差（每个新增 EVOLUTION 条目 / 每个行为变化），用大白话讲三件事：

   - **变了什么**：一句话。
   - **对你日常的影响**：从现在起你（和我）会有什么不同？要**开始 / 停止 / 改变**做什么？——这是简报的核心，必须落到行为，不能只描述功能。
   - **你的数据被怎么动了**（若有迁移）：动了哪张表 / 加了什么列 / 分区，**原内容一字未改**。

   范例（#22）：

   ```
   📌 Epic 完结后自动下沉
   · 变了什么：完结的专项不再赖在顶部，整块沉到归档前；Epic 表新增「完成」列。
   · 对你的影响：打开 index 第一眼就是"现在能做什么"，你不用再手动略过一堆绿表。
   · 你的数据：「Space 个人化模型落地」已全绿 → 整块迁到 ## 已完结专项；TODO 内容一字未动。
   ```

   收尾**问一句**"这些变化跟你现在的习惯有冲突吗、认可吗"——升级是双向对齐心智，不是单向通知。

### 上游有、本项目没有的结构怎么办（**别把升级变成问卷**）

升级常遇到"真相源有某文件 / 分区，本项目当初没建"。原则：**能合理默认 + 简报说明的，就默认，别逐个问用户**——问得越多越像考试，徒增困惑。

- **`projects.md`（工作区拓扑）**：单仓项目当初可能没建。**直接建好**（复制模板并把 `{{PROJECTS_TABLE}}` 填成一行 = 本仓本体），**不要问"要不要建"**。它是标准槽位，单仓留一行零成本、多仓时已就位。简报里一句话说明"补了 projects.md，单仓就一行"。
- **类型门控的域（PRD / EXP）**：尊重项目当初的类型——工具/库项目没有 PRD/EXP 是**故意的**，升级**不要**强行建。上游若在这些域加了新能力，简报里提一句"以后想用可补"，不创建。
- **真正该停下问用户的，只有**：数据迁移有歧义、或某变更会改动语义。其余一律默认 + 简报告知。

### 升级红线

- **绝不用真相源空 `index.md` 覆盖项目 `index.md`** —— 这会清空所有 TODO/PRD/EXP，是本操作最大事故源。
- **数据迁移只改结构不改语义** —— 加列 / 分区可以，删条目 / 改内容不行。
- **不自动 commit** —— 升级完只改文件，等用户 diff 审过再拍板提交。
- **拿不准某条 EVOLUTION 怎么迁 → 停下问用户**，不要猜着改数据。
- **同步完文件 ≠ 升级完** —— 不出 step 9 的「心智简报」、不让用户知道"日常怎么变"，这次升级就是失败的。文件是手段，心智对齐才是目的。
- **覆盖完不重读 = 拿旧脑子干新活** —— step 4 必做：bootstrap 等被覆盖的 SOP 要重新 Read 进上下文，尤其任务中途升级（启动协议不会自动重注入）。
- **别把升级变成功能问卷** —— 上游有、本项目没有的东西，能默认 + 简报说明的就别问（如 projects.md 直接补一行）；问题只留给"有歧义的数据迁移"。

- 不接 TAPD 等外部任务系统
- 不为复用 PRD 而提前抽象（YAGNI）
