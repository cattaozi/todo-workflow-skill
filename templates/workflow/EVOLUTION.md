# .workflow/ 演进日志

> 工作流自己也是个产品，会迭代、会踩坑、会沉淀。
> 这里记录每次演进——不只是"改了什么"，更重要的是"为什么改、考虑过什么、可复用的 insight 是什么"。
>
> **更新规则**：每次对 `.workflow/` 或 `CLAUDE.md` 工作流相关部分做**结构性改动**（不是日常增删 TODO），在这里加一条记录。

## 沉淀的方法论

（继承自 `todo-workflow-skill` 上游 v1；本项目运行中如有新洞察，追加到末尾。原则会留下，具体改动会过期。）

1. **SOP 不只是文档，要是 AI 的运行时合约**——光描述"做什么"不够，要规定"AI 进来必须先 Read 我"。靠"自律"会漂移；靠"@ 自动注入"才常驻。
2. **轻量 ≠ 主观**——SOP 可以短，但状态转换的门槛必须**可验证**（DoD 是目标态，AC 是可勾选清单；两者都要）。
3. **不同时做的事拆成不同 Skill**——创建 TODO vs 推进 TODO，是两个不同活动，由不同角色、不同时机驱动，应该各自一个 SOP。
4. **Skill 用 folder + 标准主文件名（SKILL.md）**——每个 Skill 自包含包，能挂模板、示例、checklist。
5. **关键 SOP 用 `@` 导入到 CLAUDE.md**——真常驻、零依赖 AI 自律；其它 Skill 按需 Read。
6. **用户的盘问权 = 外部纪律**——SOP 是合约，用户能问"刚才走的哪一条"。
7. **沉淀方法论本身要有渠道**——本文件存在的意义。
8. **「工作量大」不是技术判断**——AI 容易把"懒得搞"包装成"不值得"。把"工作量影响排期不影响决策"写进 bootstrap 红线。
9. **可计算的字段不要存**——count / last_updated 等手工同步是 bug 工厂。索引文件只存"内容"，不存"统计"。
10. **关键决策的载体是 commit message**——除非有真正的非显然决策，否则 TODO 文件不需要单独的"关键决策"章节。
11. **小修不进 TODO**——TODO 是计划级（半天起）。改 typo / 调 padding → 直接做 + commit。
12. **Skill Read 要 cache-friendly**——session 内同一 SOP 第一次 Read 全文，后续靠红线段心理复述。
13. **数据产物的 schema 要写下来，不能现编**——index.md 类型产物必须有同目录 README.md 定义结构。AI 写入前 Read schema；纯读不用。
14. **"我在管什么"是工作流的一等公民**——工作区项目拓扑（`projects.md`）是个有 schema 的 config 槽，和 todo/prd/explore 域平级：工作流拥有「槽位 + schema + bootstrap 读它」的规矩，项目只填内容。它与 `CLAUDE.md`（怎么干活）、bootstrap（按什么流程）三者正交，所以拓扑不该耦合进 CLAUDE.md。
15. **`on_hold`（搁置）是独立状态，但要带护栏**——"已成形、但现在不具备条件"既不是可拾取的 `todo`，也不是 `abandoned`。它的价值是**把项移出可拾取池**（单列搁置表）。两道护栏防它变垃圾场：① 进 `on_hold` 必须有 DoD + **复活条件**（说不出"等什么"就回 EXP 或 abandon）；② 搁置 / 放弃必须在 index **原因列**写清理由——原因列空着不算合法状态。这条源自 Agent OS 实战自发长出的搁置区。
16. **活跃项是扁平待办池，唯一的分组是 Epic**——不要按 phase / 主题 / 月份给活跃项分组（那是过度结构化）。一堆松散的 TODO 就平铺在 ## 活跃项；只有"同一件事拆出的多个 TODO"才归一个 Epic。
17. **Epic（专项）是分组视图，不是新实体**——index 里 `## 专项` 段下每个 `### Epic：<名>` 一张表（列 = 活跃表 + `原因/备注`）。不建文件、不加状态机——Epic 状态由成员派生。配套"推进 Epic"= 按依赖序逐个推、每条门槛照常的批量驱动（不降门槛）。
18. **Epic 成员不拆散，原地改状态（整块可随完结搬家——见 #22）**——和散户 TODO 不同：散户完成/搁置会逐个迁到全局 ## 归档 / ## 搁置；Epic 成员任何状态都**留在原行只改 emoji**，done/abandoned/on_hold 的理由写进 `原因/备注` 列。这样 Epic 表就是这件事的**完整台账**，回顾时一眼看全干了哪些、哪条废了为什么——散到全局归档反而把叙事打碎。注意"不拆散"≠"整块不动"：成员始终同表，但**整张表**会在 Epic 全完结时从 ## 专项 迁到 ## 已完结专项（#22）。代价：全局 ## 归档 不再是"所有已完成"的全集（Epic 成果在各自 Epic 里），值。
19. **每张表都要钉死排序，否则模型乱排**——schema 不光管列，还得管**行序**。不指定排序，不同 session 会把同一张表排出不同顺序（尤其归档）。规则：活跃项按优先级、归档按完成日期降序、搁置按编号、Epic 内按推进顺序。见 todo/README §排序规则。
20. **意图路由的触发词要覆盖口语，不能靠 AI 猜**——同一个意图有无数说法（"完成"=关闭/收工/结束/搞定/done/close）。触发词列窄了，AI 就得"猜"该走哪个 SOP——这违背"SOP 是运行时合约"。尤其 `done` 这种带硬验证门槛的操作，触发词漏了就可能跳过验证。所以把高风险操作的口语说法尽量枚举进 bootstrap 路由表。
21. **"设计层"不是缺一个独立产物，而是分布在已有结构里**——一度被 OpenSpec 那种独立 design.md 唬住，以为我们缺设计层。其实对单人/小团队尺度：`背景/Why` 是**总纲**（为什么做 + 关键约束/取舍），`任务粒度`本身是一种设计，`关键决策/方案`段装**长存的设计理由**。关键区分：**任务是脚手架（做完即作废），决策是理由（做完仍要看）**——任务列表永远答不了"半年后为什么这么建"。所以设计理由就地写进 TODO（背景 + 可选的关键决策/方案段），**不单开文件**；真正跨多 TODO 的架构级设计才升级成 PRD（那本就是独立文件）。
22. **台账与预览要分区——完结的 Epic 必须下沉，不能赖在顶部**——`## 专项` 钉死在 `## 活跃项` 正下方，初衷是"当前要按顺序推的事"靠前。但 Epic 全完结后整块还赖在顶部，日常一眼望去全是 🟢，反而**看不到"现在能做什么"**——台账（回顾用）挤掉了预览（决策用）。修复：拆出 `## 已完结专项`（在 `## 归档` 之前），Epic 末位成员一转 🟢/⚫ 就把**整块**迁过去（成员不拆，台账完整）。区段顺序变成 活跃项 → 专项（进行中）→ 搁置 → 已完结专项 → 归档。配套给 Epic 表补 `完成` 列（原来只有散户归档表有完成日期，Epic 成员"原地改状态"却无处记完成日，是个真空洞）。可复用 insight：**同一份索引同时服务"我现在做什么"（预览）和"这件事干了啥"（台账）两种读法；当一种把另一种挤出视野，就该按读法分区，而不是靠人脑过滤**。
23. **工作流要能自我更新——用 EVOLUTION 编号当版本号，靠代码/数据分层安全覆盖**——工作流装进多个项目后，上游一进化，下游就过时，却没有同步通道。难点是 `.workflow/` 把"上游 SOP"和"项目数据"混在一个目录。解法两件：① **代码/数据分层**——`README.md` + 整个 `skill/` 树是代码（可整体覆盖），`index.md` + 带编号详情 + `projects.md` 是数据（绝不覆盖，最大事故源是拿上游空 index 盖掉项目 index）；② **EVOLUTION 编号即版本号**——「沉淀的方法论」列表的最大编号就是版本，上游与项目的编号差**既标识版本、又逐条充当数据迁移指南**（每条结构性变更本就写清了改了什么）。③ **通道是本地真相源目录，不是远程 git**——真相源 = 维护者的本地工作区，软链到约定路径 `~/.claude/skills/todo-workflow`，永远最新；升级就是本项目和这个**本地目录**比对 + 覆盖，零网络往返（远程 git pull 对高频小升级成本太大）。升级 SOP 落在 per-project `README.md` §升级（因为读它的是项目里的 AI），bootstrap 路由表加"更新工作流"触发词；更进一步让 **bootstrap 启动时主动比对版本号**（数两个 EVOLUTION 的方法论最大编号），落后才提示、持平/无源则静默——这样不靠用户记得发起。**但主动检测只在会话启动跑**——用户常在一个长会话里连续干活，中途没有"新会话"，提示不会再冒；所以**手动触发必须能在 session 任意时刻发起（含长任务进行中）**，二者互补、缺一不可。④' **任务中途升级会让 AI 自己已加载的上下文（含启动协议 bootstrap 本身）过时**——bootstrap 是会话开头注入的，中途不重注入。所以"覆盖后重读被覆盖的 SOP/schema"必须是**升级 SOP 的显式步骤**（step 4「重新载入认知」），不能指望"启动协议提醒"（它自己就是旧的）。可复用 insight：**升级时唯一保证最新的，是"当下正在执行的升级 SOP"自身；所有"刷新认知"的动作都得挂在它上面，而非任何会话级注入的东西**。注意自举悖论：**升级触发器是随升级才装进 bootstrap 的，所以每个老项目的"第一次"升级得手动点一句（直接让 AI 读真相源 §升级），之后才自维持**。妙处：**这条机制自我传播**——下游一旦同步，看到的第一个版本差就是"#23 新增升级机制"，从此具备继续升级的能力。④ **升级的终点是更新人的心智，不是同步文件**——一开始升级 SOP 的收尾只是"报告覆盖了哪些文件、从 #N 到 #M"，但用户读完，明天用工作流的方式不会有任何不同——文件变了，**人的用法没变**，等于白升。所以收尾改成强制的**心智简报**：逐条版本差讲"变了什么 / 对你日常的影响（要开始/停止/改变做什么）/ 你的数据被怎么动"，并回头问用户认不认可。文件同步是手段，心智对齐才是目的；**"同步完文件 ≠ 升级完"写进升级红线**。可复用 insight：**任何会被复制分发的东西都要自带"回家的路"——版本标识 + 更新通道；版本标识能从已有结构（变更日志）派生就别再发明新字段（呼应 #9）；通道选"本地软链目录"而非"远程仓库"，因为同源同机时本地比对成本趋零。但最深的一条：分发的终点是接收者的"理解"，不是文件落地——任何"更新"动作都要以"对方心智是否真的变了"收尾，否则只是搬运字节**。⑤ **升级别变成功能问卷**（实测教训）——下游升级时撞到"上游有、本项目没建"的结构（如单仓项目没建 `projects.md`），AI 把它做成"要不要建？"问用户，一连几问像考试。修正：**能合理默认 + 简报说明的就默认**（projects.md 直接补一行），类型门控的域（PRD/EXP）尊重原类型不强建，**只有"有歧义的数据迁移"才停下问**。配套：projects.md 模板从"觉得多余可删"改成"留着，别删"——根上消除"下游主动删 → 升级又得问"的差异源。可复用 insight：**自动化流程里每多问一个本可默认的问题，都是把系统的不确定转嫁给用户；问题要留给"错了有代价且机器判不准"的地方**。
