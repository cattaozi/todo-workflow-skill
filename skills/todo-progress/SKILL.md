# /tf:todo-progress — 推进 TODO 的 SOP

> 本技能即"推进 TODO"的运行时合约。操作**当前项目**的 `.workflow/todo/`。配对：`/tf:todo-create`。
> 用户输入：$ARGUMENTS。前置：当前项目**没有 `.workflow/`** → 提示先 `/tf:init`，停。

## 文件位置

- 索引：`.workflow/todo/index.md`（活跃表 + 专项表 + 搁置表 + 已完结专项 + 归档表）
- 单条详情：`.workflow/todo/TODO_NNN_<短描述>.md`
- **索引格式（列序 / 列格式 / 排序 / 修改规则）**：见 `/tf:todo-create` §索引格式（那是权威 schema）。改 index 前照它，不要现编。下面只列推进时高频用到的迁表/排序要点。

## 状态全景

| 状态 | emoji | 含义 |
|---|---|---|
| `todo` | ⚪ | 未开始（可拾取） |
| `in_progress` | 🟡 | 在做 |
| `on_hold` | ⏸️ | 搁置——已成形但现在不具备条件，主动泊车，等复活条件 |
| `done` | 🟢 | 完成 |
| `abandoned` | ⚫ | 放弃 |

合法转换：

```
   todo ⇄ on_hold ──→ in_progress ──→ done
     └───────┴────────────┴────────→ abandoned
```

> `in_progress` 做到一半遇阻，也可退回 `on_hold`。
> **`on_hold` 不是垃圾场**：进它必须有 DoD + **复活条件**（"等什么才能继续"）。写不出复活条件 → 要么是 EXP（回灵感池），要么老实 `abandoned`。

## DoD vs AC 的区别（**推进时必须查 AC**）

| | 完成定义 (DoD) | 验收条件 (AC) |
|---|---|---|
| **形态** | 一两句话目标态 | 可勾选清单 |
| **谁用** | 工程师对照自己说"做完了" | 任何人拿来逐条核对 |
| **特征** | 抽象、隐含 | 具体、可观测、布尔 |

**状态门槛靠 AC 判定，不是靠"觉得做完了"**。

## 迁表规则：散户 vs Epic 成员（横切所有动作）

每个状态转换都要迁 `index.md`，**先按这条分流**，下面各动作的「index 迁移」栏只说迁哪、不重复这条：

- **散户 TODO**：按目标状态迁到对应分区——`done`/`abandoned` → 归档表；`on_hold` → 搁置表；唤回 → 活跃表。
- **Epic 成员（不拆散）**：任何状态都**留在自己的 `### Epic：<名>` 表**，只**原行改 emoji + 填 `完成`（done/abandoned 填日期）+ `原因/备注` 列**（🟢 选填产出 / ⚫ 必填放弃理由 / ⏸️ 必填复活条件）。
- **Epic 表整块搬家**：当**最后一个成员**转 🟢/⚫，把**整个 Epic 块从 `## 专项` 迁到 `## 已完结专项`**（在 `## 归档` 之前，成员行不拆）。目的：`## 专项` 只留在做的。

> 列序 / 列格式 / 排序规则的权威 schema 见 `/tf:todo-create` §索引格式，改 index 前照它。

## 状态转换

> 每个动作按固定五栏读：**触发 → 门槛 → 状态写入 → index 迁移 → 必填项**。门槛是布尔闸，不过就停。

### todo → in_progress（开始做）

- **触发**：用户说"做 / 开始 #NN"。
- **门槛**（两道，缺一不可）：
  1. **AC 已写好**——仍是 TBD / 不存在 → **拒绝开始**，回头走 `/tf:todo-create` 让 PM 补 AC。
  2. **写落地计划、对齐方向**（不能跳）——开工简报不是口头说说，要**落库成施工图**：
     - 先**通读整条 TODO**：背景/Why、任务、DoD、AC、关键决策/方案，都读，不只扫标题。
     - **把实现思路写进详情 `## 落地计划`**：实现路径 / 关键步骤（带顺序）/ 要动哪些文件 / 取舍点。这段**既是持久施工图（换会话 / 中断可续），也是下面简报的底稿**。
     - 拿它给用户**简短简报**：**我会做什么**（落地计划要点，不是复述任务清单）+ **做完你会得到什么**（对应 DoD / AC 的成果）。
     - **等用户认可**（或按反馈改落地计划）再动手——写代码前把"我的理解"对齐"你的意图"。简单 TODO 计划可以很短，但不能省、不能只活在脑子里。
- **状态写入**：frontmatter `status: in_progress`；落地计划已在门槛 2 落库（`## 落地计划`），执行中随进展更新 / 勾掉。
- **index 迁移**：状态列 → 🟡（散户留活跃表）。
- **必填项**：`## 落地计划` 已写入并与用户对齐。

### in_progress → done（完成）

- **触发**：用户说"#NN 做完了 / 完成 / 搞定"。
- **门槛**（两道，缺一不可）：
  1. **AC 全部勾完**——严格，任何一条 `[ ]` 都不能转 done。
  2. **验证通过**——按下面 checklist 跑完，**不因「工作量大」放弃任何一条**（见 bootstrap §Step 0）：
     - [ ] 后端动了：`<test-cmd>` / `<lint-cmd>` / `<type-cmd>` 通过
     - [ ] 前端动了：`<type-check-cmd>` / `<lint-cmd>` / `<build-cmd>` 通过
     - [ ] 用户可见改动：**需要重启服务才能验收的，必须实际重启**（user 拍板那刻看到的必须是新代码）
     - [ ] 涉及 API：curl 端到端跑通主路径（不止单测）

     > `<*-cmd>` 占位符的实际命令见本项目配置（`projects.md` / bootstrap），不在本 SOP 写死。
- **状态写入**：frontmatter `status: done` + `completed: <YYYY-MM-DD>`。
- **index 迁移**：散户迁归档表，**按完成日期降序插到正确位置**。
- **必填项**：执行中冒出**非显然决策** → 补进详情 `## 关键决策 / 方案`（长存的设计理由）。

### → on_hold（搁置）

- **触发**：用户说"搁置 #NN"，或 `in_progress` 做到一半遇阻、主动泊车。
- **门槛**：有 DoD + **明确复活条件**（不要求 AC）。写不出复活条件 → 这不是搁置：要么是 EXP（回灵感池），要么老实 `abandoned`。
- **状态写入**：frontmatter `status: on_hold`；`note` 写**为什么搁置 + 复活条件**。
- **index 迁移**：散户从活跃表移到搁置表。
- **必填项**：**原因列**填 note（为什么搁置 + 复活条件）。

### on_hold → 唤回（todo / in_progress）

- **触发**：复活条件满足，要继续这条。
- **门槛**：若直接开做（→ `in_progress`），**仍走"开始做"的两道门槛**（AC + 开工简报）。
- **状态写入**：frontmatter `status` 改回 `todo` / `in_progress`。
- **index 迁移**：散户从搁置表移回活跃表。
- **必填项**：无。

### → abandoned（放弃）

- **触发**：用户说"放弃 #NN"，或确认这条不再做（可从 `todo` / `in_progress` / `on_hold` 任一态进入）。
- **门槛**：无。
- **状态写入**：frontmatter `status: abandoned` + `completed: <放弃日期>`（放弃时 `completed` 字段复用为放弃日期）；`note` 写**为什么放弃**。
- **index 迁移**：散户迁归档表。
- **必填项**：**原因列**填放弃理由。

## 推进整个 Epic（专项）

用户说"**推进 Epic：<名>**"——这是"**带顺序的批量驱动**"，**不降低单条门槛**：

1. **读成员清单**：index `## 专项` 下 `### Epic：<名>` 表，行序 = 推进顺序。
2. **算就绪顺序**：跳过 `依赖` 列里还没 done 的前置项；先推无未完成依赖的成员。
3. **逐个推**：每个就绪成员走上面的标准转换（`todo → in_progress → done`），**每步门槛照常**。
4. **遇阻即停、不硬推**：缺 AC / 撞到需用户拍板 / 验证不过 → **停在那条，报告进度**，等用户。
5. **每完成一个就更新 index**：按 §迁表规则——Epic 成员**原行**改状态（成员不拆）。
6. **全部 🟢/⚫** → 按 §迁表规则把整块 Epic 迁到 `## 已完结专项`，报告 Epic 收尾。

## 推进完成的 AC（**元 AC**）

每个动作收尾后自查（对应五栏骨架）：

- [ ] **状态写入**：详情 frontmatter status / completed 已更新
- [ ] **index 迁移**：位置正确——散户迁对应分区；Epic 成员**原地**改状态 + 填 `完成` + `原因/备注`
- [ ] (Epic 末位成员完结) 整块 Epic 已从 `## 专项` 迁到 `## 已完结专项`
- [ ] **必填项**：(开始做) `## 落地计划` 已写入并与用户对齐；(on_hold / abandoned) 原因列已填；(done) 关键决策已补
- [ ] (done) 验证 checklist 全过

## 收工（提交拍板）

平时**不自动 commit**；只有用户明说"**收工 / 完成 / 提交 / commit / 搞定了**"才动手——这句话**就是提交拍板**，但**只解除当下这一轮**的不自动提交（不是从此自动提交，下一轮仍要等话）。

**单仓库**：`git add <具体路径>`（**不要 `-A`**，防 `.env`/构建产物误入）→ 按项目 commit 规范提交（message 含相关 TODO 编号）。

**多仓库**（`projects.md` 列了多个独立 git 仓时）：**逐个子仓库**提交，每个：
- 有远程的 → 先 `git pull --rebase` 再提交
- 没有 upstream 的 → 跳过 pull
- ⚠️ **子仓目录绝不 `git add` 进根仓库** —— 各仓独立提交，别把子仓当根仓的文件纳进去
- 具体有哪些仓、哪些目录该排除 → 看本项目 `projects.md`（项目配置，不在本 SOP 里写死）

**`pull --rebase` 撞上大分叉**（本地与远程已严重分叉、逐个 replay 旧提交会反复冲突、有丢逻辑风险）：
- **别硬 replay** → 先把**本轮工作 commit 保命** → 改用一次性 `merge` 解 → **解完必须验证**（跑测试/构建，确认 merge 没弄坏）

**职责边界**：**commit 我做，push 归你拍**——提交完**不要自动 push**。提交完**报告**：每个仓的 commit 清单 + 验证证据（测试/构建结果）。

## 红线

- **没有 AC 不能转 in_progress** —— 缺了回头走 `/tf:todo-create`
- **不写落地计划就闷头写代码 = 方向跑偏的头号来源** —— 转 in_progress 前必须通读全 TODO + 把实现思路写进 `## 落地计划` + 与用户对齐；计划要落库，不能只口头说（见 §todo → in_progress 门槛 2）
- **没勾完 AC 不能转 done** —— 不能自己说"差不多了"
- **AC 不能临场放水** —— 改 AC 内容（甚至拆新 TODO），不能编理由打勾
- **不要跨级跳** —— 不要 todo → done 跳过 in_progress
- **`on_hold` 必须带复活条件**；**搁置 / 放弃必须写原因**
- **不要重排编号**
- **不自动 commit、不自动 push** —— 平时只改文件；"收工/提交"是提交拍板（仅当下这轮），push 永远等用户。具体编排见上方 §收工
- **「工作量大」不能让你少跑验证** —— 见 bootstrap §Step 0
