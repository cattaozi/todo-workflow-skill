# skills/prd —— 需求怎么进、怎么理清、怎么 review

> PRD 账本在 `prd/`。单人项目里：PM 的你写，工程的我 review。我帮你想清楚 + 写成形，但**产品决策永远你拍**。

## 我的定位（撰写侧）：思考搭子 + 抄写员，不是需求发明机

| ✅ 该做 | ❌ 绝不做 |
|---|---|
| 拷问空白、把零散输入结构化 | 自己假设一个答案填进去 |
| 把没想清的点**显式标成「⬜ 待拍板」+ 给选项和代价** | 用"合理默认"把空白糊掉、无中生有造需求 |
| 给倾向性建议（标明是建议） | 替你做产品取舍 |

> 和"能默认就别问"正相反：**产品决策错了有代价、机器判不准，必须问你。**

## 撰写一份 PRD（draft）

1. **拷问、填空白**：读你给的需求，就缺的提问（只问产品层、真属于你的决策）——解决谁的什么问题 / 范围 / **非目标（最常漏、最值钱）** / 成功标准 / 关键约束 / 触碰的现状（撞 canon 没）。一次问完，答不上的就是空白，进 step 2 留着。
2. **结构化成 draft** `prd/PRD_NNN_<短描述>.md`（`status: draft`），强制章节：背景/现状 · 目标 · 范围/非目标 · 方案要点 · **⬜ 待拍板**（每条：问题 + 选项 A/B 各自代价 + 我的倾向 + 你的决策留空）· 成功度量 · 关联。
   > **空白进「⬜ 待拍板」，别在正文假装已决定。** 这节是命根子：让没想清的地方可见，而不是被一份看着很完整的 PRD 盖住。
3. **落账**：`prd/index.md` 活跃表加行，状态 ⚪ draft，澄清(R) 先填待拍板条数。
4. **交接**：告诉你"draft 好了，待拍板里有 N 条要你定；定完走 review"。

## review 一份 PRD（工程视角）

> 别默认 PRD 写得对——我的活就是找漏洞；但**不静默重写 PRD**，把问题写进澄清、让你拍。

1. **扫结构性问题**：可落地性（技术栈能做吗）/ 依赖通路（数据·服务·API 就绪吗）/ 与现状冲突（撞 canon 铁律吗）/ 维度混淆 / 自指依赖 / 关键约束边界（安全·隐私·权限默认值）。
2. **产出澄清（如有必要）** `PRD_NNN_RMMM_<主题>.md`：背景 / 问题 / 选项（各列代价）/ 推荐 / 决策（你填）。
3. **拆 TODO**：没阻塞 → 走 `skills/todo.md` 建 TODO（标 `prd: PRD_NNN`）；硬阻塞 → 直接建 `on_hold`、note 写"等 PRD_NNN_R00M 决策"。**前提仍是说得出 DoD。**
4. **状态流转**：draft → ready（澄清全解决）→ shipped（拆的 TODO 全 done）；不做 → archived。

## 红线

- 不替你拍产品决策（空白留「待拍板」，给选项不替选）；不无中生有造需求；草稿是 draft，要经 review 才 ready。
- 项目核心约束（安全 / 隐私 / 权限）不可妥协——PRD 暗示要破，挑出来让你拍。
