# skills/prd —— 我怎么处理产品功能描述

> PRD 是对一个产品功能、产品变更或用户问题解决方案的正式描述。它不是任务本身；它先把产品方案说清，再被解读和拆成 TODO。

## PRD 模型

- **PRD**：产品功能描述，记录背景、目标、范围、非目标、方案、待拍板、成功标准和关联。
- **创建 PRD**：你表达“创建一个 PRD / 设计一个功能 / 把这个需求写清楚”时，我和你共创 PRD；这一步要多引导。
- **解读 PRD**：我对 PRD 做对抗式理解，找不合理、不完整、不可落地和自相矛盾之处，产出解读文件。
- **任务拆解**：把已解读的 PRD 拆成 TODO；通常拆成 Epic，因为一个 TODO 一般干不完一个 PRD。
- **index**：`prd/index.md`，只记录 PRD 状态、解读数量和备注。

## 处理入口

拿到需求相关输入，按顺序判断：

1. **创建场景**：你要创建 PRD，或我判断某个功能需要先定义产品方案。
2. **解读场景**：你给了已有 PRD，或要求 review / 解读 / 看看是否合理。
3. **拆解场景**：你要求拆任务，或 PRD 已完成解读并准备进入开发。
4. **已有编号**：有 PRD 编号就读取对应 PRD、解读文件和 index；没有编号才创建新 PRD。
5. **进入对应动作**：创建 PRD、解读 PRD、任务拆解。

## 核心规则

### 我的定位

我做：

- 引导你把产品功能说清楚。
- 把零散输入结构化成 PRD。
- 暴露产品空白和决策点。
- 对抗式解读 PRD，找漏洞。
- 把 PRD 拆成 Epic / TODO。

我不做：

- 不替你做产品取舍。
- 不用“合理默认”填掉产品空白。
- 不把未决策内容写成已确定方案。
- 不静默改 PRD 来掩盖问题。

### 状态

PRD 文件 frontmatter 和 `prd/index.md` 使用同一状态：

```text
draft → reviewed → decomposed
任意状态 → archived
```

状态含义：

- `draft`：PRD 草稿，可能仍有待拍板。
- `reviewed`：已完成至少一次对抗式解读，并产出解读文件。
- `decomposed`：已拆成 Epic / TODO。
- `archived`：明确不做或停止推进。

## 动作协议

### 创建 PRD

门槛：

- 用户表达要创建 PRD、设计功能、整理需求。
- 或我判断当前输入已经超过 EXP，进入产品功能定义更合适。

动作：

1. 分配下一个 `PRD_NNN`：扫描 `prd/` 下已有编号，取最大编号加一；没有就从 `PRD_001` 开始。
2. 创建 `prd/PRD_NNN.md`。
3. 引导你补齐产品层信息：用户 / 问题 / 目标 / 范围 / 非目标 / 成功标准 / 约束。
4. 不确定或需要你拍板的内容写进 `## 待拍板`。
5. 写入 `prd/index.md` 活跃表：编号、标题、状态 `draft`、解读数、备注。

PRD 文件结构：

```md
---
status: draft
created: <YYYY-MM-DD>
completed:
note:
---

# PRD_NNN <标题>

## 背景 / 现状

## 用户与问题

## 目标

## 范围

## 非目标

## 方案要点

## 待拍板

## 成功标准

## 关联
```

待拍板写法：

```md
### <问题>

- 选项 A：
- 代价：
- 选项 B：
- 代价：
- 我的倾向：
- 你的决策：
```

`你的决策` 为空时，相关内容不能写成已确定方案。

### 解读 PRD

门槛：

- 已有 PRD 文件。
- 用户要求解读、review、找问题，或 PRD 准备拆 TODO。

动作：

1. 通读 PRD、已有解读文件、`projects/memory.md`。
2. 对抗式检查：目标是否真实、范围是否清楚、非目标是否缺失、方案是否自洽、依赖是否存在、约束是否冲突、验收是否可观察。
3. 产出解读文件 `prd/PRD_NNN_R0001.md`；同一 PRD 多次解读时编号递增。
4. 更新 `prd/index.md` 解读数。
5. PRD frontmatter 和 index 状态改为 `reviewed`。

解读文件结构：

```md
# PRD_NNN_R0001 <主题>

## 我的理解

## 不合理之处

## 缺失信息

## 风险与依赖

## 建议调整

## 需要你拍板
```

解读文件只记录我的对抗式理解和问题，不直接改写 PRD 正文。

### 任务拆解

门槛：

- PRD 状态是 `reviewed`。
- 阻塞拆解的待拍板已经有决策。
- 能拆出说得出 DoD 的工程任务。

动作：

1. 先判断是否需要 Epic；默认 PRD 拆成 Epic，因为一个 TODO 一般干不完整个 PRD。
2. 创建 Epic，并把 PRD 关联写入 Epic README 或成员 TODO 关联中。
3. 按 `skills/todo.md` 创建成员 TODO。
4. 如果某个 TODO 依赖未决策或外部条件，创建为 `on_hold`，备注写阻塞来源。
5. PRD frontmatter 和 index 状态改为 `decomposed`。

### 归档 PRD

动作：

1. PRD frontmatter 状态改为 `archived`。
2. `note` 写不做或停止推进的原因。
3. 从 `prd/index.md` 活跃表迁到归档表。

## 持久化规则

### index 写法

- index 是台账，放状态、解读数量和备注。
- PRD 文件是真相，放产品功能描述和拍板结果。
- 解读文件是评审产物，放我的对抗式理解，不替代 PRD 正文。

### `prd/index.md` 初始结构

```md
# PRD

## 活跃

| 编号 | 标题 | 状态 | 解读 | 备注 |
|---|---|---|---|---|

## 归档

| 编号 | 标题 | 结果 | 日期 | 备注 |
|---|---|---|---|---|
```

## 红线

- 不替你拍产品决策。
- 产品空白必须进 `## 待拍板`。
- `你的决策` 为空时，不能把相关方案写成已确定。
- 未解读的 PRD 不直接拆 TODO。
- 不无中生有造需求。
- 安全、隐私、权限边界不清时必须写入解读文件。
