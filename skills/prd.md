# skills/prd —— 我怎么处理产品功能描述

> PRD 是对一个产品功能、产品变更或用户问题解决方案的正式描述。它不是任务本身；它先把产品方案说清，再被解读和拆成 TODO。

## PRD 模型

- **PRD**：产品功能描述，记录背景、目标、范围、非目标、方案、待拍板、成功标准和关联。
- **创建 PRD**：用户语义是在把产品功能、产品变更或需求方案写成正式 PRD 时，我和你共创 PRD；这一步要多引导。
- **解读 PRD**：我对 PRD 做对抗式理解，找不合理、不完整、不可落地和自相矛盾之处，产出解读文件。
- **产品建议稿**：当我们对某个产品功能有不同理解，需要写给产品经理讨论或拍板时，产出面向产品经理的建议稿。它不是 PRD，不直接进入开发。
- **任务拆解**：把已解读的 PRD 拆成 TODO；通常拆成 Epic，因为一个 TODO 一般干不完一个 PRD。
- **index**：`prd/index.md`，索引 `prd/` 目录下的 PRD 和产品建议稿；解读文件只计入对应 PRD 的解读数量。

## 处理入口

拿到需求相关输入，先判断用户是在讨论、让我理解，还是在要求产出 PRD 相关文档。

1. **讨论 / 理解语境**：用户只是讨论需求、校准理解或探索方向时，先回应讨论，不自动创建 PRD 文件。
2. **创建 PRD 语境**：用户语义是在形成正式产品功能描述时，进入创建 PRD 动作。
3. **解读 PRD 语境**：用户提供已有 PRD，并希望我评审、找问题、判断合理性或准备拆解时，进入解读 PRD 动作。
4. **产品建议稿语境**：用户语义是在形成给产品经理讨论或拍板的材料时，进入产品建议稿动作。
5. **任务拆解语境**：用户语义是在把已明确的 PRD 转成工程任务时，进入任务拆解动作。
6. **已有对象**：输入指向具体 PRD 编号、建议稿编号或文件时，先读取对应文件和 `prd/index.md`。

## 核心规则

### 我的定位

我做：

- 引导你把产品功能说清楚。
- 把零散输入结构化成 PRD。
- 把功能理解分歧写成给产品经理看的产品建议稿。
- 暴露产品空白和决策点。
- 对抗式解读 PRD，找漏洞。
- 把 PRD 拆成 Epic / TODO。

我不做：

- 不替你做产品取舍。
- 不用“合理默认”填掉产品空白。
- 不把未决策内容写成已确定方案。
- 不静默改 PRD 来掩盖问题。

### 状态

PRD 文件 frontmatter 使用文本状态：

```text
draft → reviewed → decomposed
任意状态 → archived
```

状态含义：

- `draft`：PRD 草稿，可能仍有待拍板。
- `reviewed`：已完成至少一次对抗式解读，并产出解读文件。
- `decomposed`：已拆成 Epic / TODO。
- `archived`：明确不做或停止推进。

PRD index 状态 emoji：📝 draft，🔍 reviewed，🧩 decomposed，📦 archived。

产品建议稿 frontmatter 使用文本状态：

```text
draft → submitted → accepted / rejected / converted / archived
```

状态含义：

- `draft`：还在整理。
- `submitted`：已给产品经理或准备用于讨论。
- `accepted`：建议被采纳。
- `rejected`：建议未采纳。
- `converted`：已转成正式 PRD。
- `archived`：停止推进。

产品建议稿 index 状态 emoji：📝 draft，📨 submitted，✅ accepted，❌ rejected，🔁 converted，📦 archived。

## 动作协议

### 创建 PRD

门槛：

- 用户语义明确是在形成正式产品功能描述。
- 当前输入更适合产品功能定义但用户尚未要求建 PRD 时，只建议进入 PRD，等用户确认。

动作：

1. 分配下一个 `PRD_NNN`：扫描 `prd/` 下已有编号，取最大编号加一；没有就从 `PRD_001` 开始。
2. 创建 `prd/PRD_NNN.md`。
3. 引导你补齐产品层信息：用户 / 问题 / 目标 / 范围 / 非目标 / 成功标准 / 约束。
4. 不确定或需要你拍板的内容写进 `## 待拍板`。
5. 写入 `prd/index.md` 的“PRD”表：编号、标题、状态 📝、解读数、备注。
6. PRD 由用户原始材料生成时，先确保原件或可追溯副本进入 `inbox/`，再在标题后插入 `## 来源 / Source` 并写明生成依据。只有当前对话指令时，不创建来源段。

PRD 文件结构：

```md
---
status: draft
created: <YYYY-MM-DD>
completed:
note:
---

# PRD_NNN <标题>

## 背景 / 现状

## 用户与问题

## 目标

## 范围

## 非目标

## 方案要点

## 待拍板

## 成功标准

## 关联
```

待拍板写法：

```md
### <问题>

- 选项 A：
- 代价：
- 选项 B：
- 代价：
- 我的倾向：
- 你的决策：
```

`你的决策` 为空时，相关内容不能写成已确定方案。

### 解读 PRD

门槛：

- 已有 PRD 文件。
- 用户语义是在要求解读、评审、找问题，或准备把 PRD 拆成工程任务。

动作：

1. 通读 PRD、已有解读文件、`projects/README.md`。
2. 对抗式检查：目标是否真实、范围是否清楚、非目标是否缺失、方案是否自洽、依赖是否存在、约束是否冲突、验收是否可观察。
3. 产出解读文件 `prd/PRD_NNN_R0001.md`；同一 PRD 多次解读时编号递增。
4. 更新 `prd/index.md` 解读数。
5. PRD frontmatter 状态改为 `reviewed`，index 状态改为 🔍。

解读文件结构：

```md
# PRD_NNN_R0001 <主题>

## 我的理解

## 不合理之处

## 缺失信息

## 风险与依赖

## 建议调整

## 需要你拍板
```

解读文件只记录我的对抗式理解和问题，不直接改写 PRD 正文。

### 创建产品建议稿

门槛：

- 用户语义是在形成给产品经理讨论或拍板的材料。
- 我们对某个产品功能、产品行为、范围、目标或方案有不同理解，需要形成讨论材料。
- 当前内容还不是已拍板 PRD，也不应直接进入 TODO。

动作：

1. 分配下一个 `PRD_PROPOSAL_NNNN`：扫描 `prd/` 下已有建议稿编号，取最大编号加一；没有就从 `PRD_PROPOSAL_0001` 开始。
2. 创建 `prd/PRD_PROPOSAL_NNNN.md`。
3. 写清：背景、当前理解分歧、我的判断、建议方案、取舍依据、给产品经理的问题、建议下一步。
4. 如果建议稿针对已有 PRD，在 frontmatter 和正文关联里写明 PRD 编号；没有关联 PRD 就留空。
5. 写入 `prd/index.md` 的“产品建议稿”表：编号、标题、状态 📝、关联、备注。
6. 建议稿由用户原始材料生成时，先确保原件或可追溯副本进入 `inbox/`，再在标题后插入 `## 来源 / Source` 并写明生成依据。只有当前对话指令时，不创建来源段。

产品建议稿文件结构：

```md
---
status: draft
created: <YYYY-MM-DD>
submitted:
related:
note:
---

# PRD_PROPOSAL_NNNN <标题>

## 背景

## 当前理解分歧

## 我的判断

## 建议方案

## 取舍依据

## 给产品经理的待确认问题

## 建议下一步

## 关联
```

产品建议稿不是 PRD。它被采纳后，可以转成 PRD；被驳回或搁置时，只更新建议稿状态和 index，不拆 TODO。

### 任务拆解

门槛：

- PRD 状态是 `reviewed`。
- 阻塞拆解的待拍板已经有决策。
- 能拆出目标、范围、规格和 AC 清楚的任务级实施规格。

动作：

1. 先判断是否需要 Epic；默认 PRD 拆成 Epic，因为一个 TODO 一般干不完整个 PRD。
2. 创建 Epic，并把 PRD 关联写入 `todo/epic/<slug>.md` 或成员 TODO 关联中。
3. 按 `skills/todo.md` 创建成员 TODO。
4. 外部条件只阻塞实施、不影响规格完整性时，按 `skills/todo.md` 创建 TODO 后改为 `on_hold`，备注写阻塞来源；未决策会改变目标、范围、规格或 AC 时，先保留为 PRD 待拍板，不创建 TODO。
5. PRD frontmatter 状态改为 `decomposed`，index 状态改为 🧩。

### 更新产品建议稿状态

门槛：

- 已有产品建议稿。
- 用户语义是在记录建议稿已提交、被采纳、被驳回、转成 PRD 或停止推进。

动作：

1. 读取建议稿和 `prd/index.md` 对应行。
2. 按事实更新建议稿 frontmatter 状态和 `prd/index.md` 状态。
3. `submitted` 补 `submitted: <YYYY-MM-DD>`。
4. `accepted`、`rejected`、`archived` 在 `note` 写清结果或原因。
5. `converted` 必须关联新建或已有 PRD 编号；没有 PRD 编号时，先创建 PRD 或停下说明缺口。
6. 只更新建议稿状态，不直接拆 TODO。

### 归档 PRD

动作：

1. PRD frontmatter 状态改为 `archived`。
2. `note` 写不做或停止推进的原因。
3. 从 `prd/index.md` 活跃表迁到归档表。

### 归档产品建议稿

动作：

1. 建议稿 frontmatter 状态改为 `archived`。
2. `note` 写停止推进的原因。
3. 从 `prd/index.md` 的“产品建议稿”表迁到归档表。

## 持久化规则

### index 写法

- index 是 `prd/` 目录台账，按产物类型分区。
- PRD 文件是真相，放产品功能描述和拍板结果。
- 解读文件是评审产物，放我的对抗式理解，不替代 PRD 正文。
- 产品建议稿是讨论材料，放功能理解分歧、建议方案和给产品经理的待确认问题。
- 解读文件不单独进 index，只计入对应 PRD 的解读数量。
- 详情文件 frontmatter 的 `status` 使用文本；index 的“状态”列只写 emoji。

### `prd/index.md` 初始结构

```md
# PRD

## PRD

| 编号 | 标题 | 状态 | 解读 | 备注 |
|---|---|---|---|---|

## 产品建议稿

| 编号 | 标题 | 状态 | 关联 | 备注 |
|---|---|---|---|---|

## 归档

| 类型 | 编号 | 标题 | 结果 | 日期 | 备注 |
|---|---|---|---|---|---|
```

## 红线

- 不替你拍产品决策。
- 产品空白必须进 `## 待拍板`。
- `你的决策` 为空时，不能把相关方案写成已确定。
- 未解读的 PRD 不直接拆 TODO。
- 产品建议稿不直接拆 TODO；只有转成 PRD 后才进入拆解。
- 不无中生有造需求。
- 安全、隐私、权限边界不清时必须写入解读文件。
