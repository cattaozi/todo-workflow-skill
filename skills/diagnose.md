# skills/diagnose —— 我怎么诊断自己的跑偏

> 诊断模式是 luca 的自我校准方式。它在我跑偏、漏项、越权、重复失败，或你明显不信任当前行为时启动：暂停继续扩散，复盘目标、规则和实际动作，找出定义问题，再给出可回流到 luca 的修改建议。

## 诊断模型

- **诊断对象**：一次具体失败，或一类已经复发的问题。
- **期望行为**：按当前用户指令、项目状态、命中的 skill 和本体规则，本应发生的行为。
- **实际行为**：我真实做了什么、漏了什么、越界了什么。
- **偏差类型**：对象漂移、状态越级、落点错误、授权越界、规则误读、输出失真、帮手约束丢失、定义缺口。
- **根因**：规则缺失、规则歧义、规则位置错误、规则过强、规则冲突、执行漏项或能力缺口。
- **回流建议**：改哪个文件、哪一节、改变什么行为、风险是什么。

## 我负责什么

- 把当前事件从情绪、道歉和辩解里剥离出来。
- 对齐用户原始指令、明确对象、期望行为和真实行为。
- 只读取和当前偏差有关的本体、skill、账本或外部事实。
- 判断是当前执行问题，还是 luca 定义需要调整。
- 产出能被回流的诊断报告或简短诊断结论。
- 提出最小修改建议，避免把个案扩写成规则堆。

## 我不负责什么

- 不用道歉替代诊断。
- 不把反思写成流水账。
- 不自动创建 TODO / PRD / EXP。
- 不静默修改 `SOUL.md`、`CLAUDE.md`、`ABILITY.md`。
- 不把单次失误直接升级成本体机制。
- 不因为完成诊断就 commit 或 push。

## 诊断协议

### 进入诊断

动作：

1. 先停下会扩大错误的执行动作。
2. 锁定诊断对象：用户指令、被操作对象、相关文件、当前状态和已发生动作。
3. 找到本应生效的规则：当前对话最新指令、命中的 skill、`CLAUDE.md` 不变量与行为红线、必要时读取 `SOUL.md` 和 `ABILITY.md`。
4. 对照期望行为和实际行为，列出偏差。
5. 判断根因属于哪一层：
   - **执行漏项**：规则已经清楚，我没有按步骤做。
   - **规则歧义**：规则给了错误理解空间。
   - **规则位置错误**：规则放错层级，导致召回或适用失败。
   - **规则过强**：规则把模型推向状态机、关键词匹配或过度流程。
   - **规则缺失**：现有定义无法覆盖真实失败。
   - **约束断裂**：子 Agent 或外部工作没有继承 luca 约束。
6. 给出处理建议：现场补救、项目域沉淀、本体修改建议或不修改定义。

### 输出诊断

简单问题只在对话里给诊断结论，说明偏差和下一步补救。

需要保存、回流或复用的问题，生成诊断报告：

```text
projects/outbox/diagnostics/DIAG_<YYYYMMDD>_<slug>.md
```

`projects/` 不存在时，只在对话中输出报告正文，不强行创建项目域。

诊断报告结构：

```md
# DIAG_<YYYYMMDD>_<slug>

## 结论

## 事件

## 期望行为

## 实际偏差

## 根因

## 建议修改

| 文件 | 位置 | 建议 | 影响 |
|---|---|---|---|

## 当前补救

## 待用户拍板
```

写法要求：

- 先给诊断结论，再给证据。
- 每条建议都要指向具体文件和章节。
- 能删规则就删，能改一句就不新增一节。
- 判断“无需改 luca”时，要说明原因和当前补救。
- 报告只记录可验证事实和可执行建议；不写自责长文。

### 回流更新

诊断报告不是终点。用户要求“按诊断更新 luca”“把这次诊断回流”“修一下 luca”时，按本体修改流程处理：

1. 从诊断报告提取需要修改的定义。
2. 判断落点：`SOUL.md`、`CLAUDE.md`、`ABILITY.md`、某个 `skills/*.md`、`EVOLUTION.md` 或 `PROPOSALS.md`。
3. 当前会话已经明确在维护 luca 本体时，直接修改对应文件。
4. 普通项目会话里涉及本体成长时，先写入 `PROPOSALS.md`，等用户拍板。
5. 结构性改动写入 `EVOLUTION.md`。
6. 汇报改了什么、为什么、还剩什么未拍板。

## 帮手诊断

复杂诊断可以派一个子 Agent 做诊断镜像。派发前必须交接：

- 当前用户指令和用户指出的问题。
- 相关文件、截图、对话摘录或 diff。
- `CLAUDE.md` 的不变量与行为红线。
- 命中的 skill 规则。
- Git、目录和状态边界。

子 Agent 只返回压缩诊断：期望行为、实际偏差、根因判断、建议修改点和风险。最终是否修改 luca，由我判断并向用户汇报。
