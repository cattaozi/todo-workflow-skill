# /tf:todo-create — 创建 TODO 的 SOP

> 本技能即"创建 TODO"的运行时合约，操作**当前项目**的 `.workflow/todo/`。配对：`/tf:todo-progress`（推进）。
> 前置：项目**没有 `.workflow/`** → 提示先 `/tf:init`，停。用户输入：$ARGUMENTS

## TODO 的边界（**先想清再开**）

TODO 是**计划级**轨道——值得"开始 → 推进 → 验收"的一件事。

- ✅ "实现登录后端"、"接入 markdown 编辑器"
- ❌ "改个变量名 / 调 padding / 修 typo"——直接做，不走流程
- ❌ "整理一下代码"——目标太虚

判定：**说不出 DoD 就不该开 TODO**；说得出但 < 半天 → 也不开（直接做 + commit）。

## 单条详情文件

- 位置：`.workflow/todo/TODO_NNN_<英文短描述>.md`（编号 3 位，**永不复用**，含 done/abandoned）
- frontmatter：

```yaml
---
id: NN
title: "<标题>"
status: todo            # todo | in_progress | on_hold | done | abandoned
added: "YYYY-MM-DD"
completed: null
prd: "PRD_001"          # 可选：从哪个 PRD 拆来
explore: "EXP_001"      # 可选：从哪个预研转来
epic: "<Epic 名>"       # 可选：属于哪个 Epic（值 = 索引 ### Epic：后的名字，逐字一致）
priority: P0            # 可选：P0 | P1 | P2
note: "一句话当前进展"   # on_hold/abandoned 时 = 原因（搁置含复活条件）
---
```

- **强制章节**：`## 背景 / Why`（整条总纲：为什么做 + 关键约束/取舍）、`## 任务`（checkbox）、`## 完成定义（DoD）`、`## 验收条件（AC）`（可暂 TBD）、`## 关联`
- **可选**：`## 关键决策 / 方案`（只在有"非显然、未来会被问为什么这么建"的长存设计理由时写）

### 详情文件模板（复制当起点）

```markdown
---
id: NN
title: "<标题>"
status: todo
added: "YYYY-MM-DD"
completed: null
priority: P0
note: ""
---

# TODO #NN — <标题>

## 背景 / Why
<!-- 总纲：为什么做 + 驱动实现的关键约束/取舍。下面任务为什么这么拆，读这节要懂 -->

## 关键决策 / 方案
<!-- 可选；没有就删。装长存的设计理由（区别于做完即作废的任务） -->

## 任务
- [ ] 子任务 1

## 完成定义（DoD）
<!-- 一两句话目标态 -->

## 验收条件（AC）
<!-- 可勾选、可观测、布尔；可暂 TBD，但进 in_progress 前必须补 -->
- [ ] AC 1

## 关联
- PRD: / 依赖: #NN / 视觉参考:
```

## DoD vs AC（**别混**）

| | DoD | AC |
|---|---|---|
| 形态 | 一两句话目标态 | 可勾选清单 |
| 谁用 | 工程师对照说"做完了" | 任何人逐条核对 |
| 特征 | 抽象、隐含 | 具体、可观测、布尔 |

## 创建步骤

1. **粒度自检**：0.5-2 天能干完？不能 → 先拆，不要硬塞。
2. **分配编号**：上一个 + 1，永不复用。
3. **创建详情文件**：完整 frontmatter + 至少 3 节（背景/任务/DoD）；AC 可暂 TBD。
4. **更新 `.workflow/todo/index.md`**：按下面「索引格式」加行，不要现编。

## 创建完成的 AC（元 AC）

- [ ] frontmatter 完整（id/title/status/added/关联字段）
- [ ] 至少 3 节：背景 + 任务 + DoD
- [ ] index 加了行：`todo` 进活跃表（属于 Epic 的进对应 `### Epic：` 表）；创建即 `on_hold` 进搁置表 + 原因列

## 红线

- 不批量建一堆不完整的；不为"将来可能做"建（那是 EXP，除非用户明说"开 TODO"）；不为 5 分钟改动建
- 不省 DoD；不重排编号；AC 创建可 TBD 但**进 in_progress 前必须补**

# 索引格式（`.workflow/todo/index.md` 的权威 schema —— 写之前照这个，`/tf:todo-progress` 也依此）

**Frontmatter** 只一行 `type: todo`，**禁存** count/last_updated（手工同步是 bug 工厂，要数总数现场数行）。

**区段顺序（钉死）**：活跃项 → 专项（进行中）→ 搁置 → 已完结专项 → 归档。活跃项 + 进行中专项居首 = 日常"现在能做什么"；Epic 全完结整块下沉到「已完结专项」，不让绿表淹没预览。

**列序（钉死，禁改）**：
- 活跃表：`编号 | 标题 | 状态 | 添加 | 优先级 | 依赖`
- 专项表（`## 专项` 与 `## 已完结专项` 共用）：`编号 | 标题 | 状态 | 添加 | 完成 | 优先级 | 依赖 | 原因/备注`（未完成成员 `完成` 填 ——）
- 搁置表：`编号 | 标题 | 状态 | 添加 | 原因（含复活条件）`
- 归档表：`编号 | 标题 | 状态 | 添加 | 完成 | 原因`

**列内格式**：编号 = `[#NNN](TODO_NNN_xxx.md)`；状态 emoji ⚪todo/🟡in_progress/⏸️on_hold/🟢done/⚫abandoned；添加/完成 = `MM-DD`（跨年用 `YYYY-MM-DD`）；优先级 P0/P1/P2，无则 `——`；依赖逗号分隔 `#NNN`，无则 `——`；原因列（搁置/归档/Epic）见下「不要做的事」必填规则。

**排序规则（钉死，禁现编）**：
- 活跃项：优先级 P0→P2 在上，同级按**编号升序**
- 专项（Epic 内）：按**推进顺序**（依赖在前的在上），不按日期/状态重排
- 搁置：**编号升序**
- 已完结专项（Epic 之间）：按 **Epic 完成日期（成员最晚完成日）降序**；Epic 内仍按推进顺序
- 归档：**完成日期降序**；`完成` 为 `——` 视作最早排最下；同日按编号降序

**Epic（专项）约定**：
- 何时开：一件事拆出 **≥3 个** TODO 且要按顺序推 → 开 Epic；少数松散相关的留活跃项
- 成员标 `epic: "<名>"`（逐字等于 `### Epic：` 后的名字）
- **成员不打散**：永远同一张 Epic 表，任何状态原行改 emoji + 填 `完成`/`原因备注`；**绝不**拆散到全局搁置/归档
- **Epic 全完结**（成员全 🟢/⚫）→ **整块**从 `## 专项` 迁到 `## 已完结专项`（成员不拆）

**修改规则**：
- 散户 TODO（不属任何 Epic）：创建→活跃项加行；→in_progress 改 🟡；→on_hold 删活跃行、搁置表加行（原因含复活条件）；→done 删活跃行、归档表加行（完成日期+原因）；→abandoned 删行、归档表加行（⚫+放弃理由）
- Epic 成员：创建→加到对应 `### Epic：` 表；任何状态流转→**原地**改 emoji + 填 `完成`/`原因备注`，行不拆；末位成员完结→整块 Epic 迁到 `## 已完结专项`

**不要做的事**：改列序 / 加列 / 给活跃项分组（要分组就开 Epic）/ 把 on_hold 留活跃表 / 把 done·abandoned 留活跃表 / 把 Epic 成员拆散到全局搁置·归档 / 完结 Epic 赖在 `## 专项` 顶部 / 搁置·归档·Epic 的原因列留空（done 可空，abandoned·on_hold 必填）/ 同一 TODO 出现两处 / 不按排序规则排 / 非 ISO 日期 / frontmatter 加 count
