# skills/bug —— 我怎么记录和修复 Bug

> Bug 是轻量修复账本。它保留问题如何表现、为什么发生、如何验证、最终怎样解决，让缺陷从发现走到消除；它不承担任务规划，也不扩展成 TODO spec。

## 账本结构

`bug/index.md` 是唯一 Bug 台账，按“待修复 / 已修复”展示当前归属。

```md
# BUG

## 待修复

| 编号 | 现象 | 创建日期 |
|---|---|---|

## 已修复

| 编号 | 现象 | 创建日期 | 结束日期 |
|---|---|---|---|
```

详情文件固定为 `bug/BUG_NNNN.md`，编号全局唯一、永不复用。

```md
# BUG_NNNN <简短标题>

## 现象

## 根因

## 测试方式

## 解决方法
```

四个字段各有明确职责：

- **现象**：记录可观察到的异常、发生条件和实际结果；需要引用原始材料时，直接在这里给出链接。
- **根因**：记录由代码、日志、复现或验证支持的因果机制；没有证据时留空，不把猜测写成根因。
- **测试方式**：记录可重复执行的验证方法；修复前能证明问题存在，修复后能证明问题消失。
- **解决方法**：只在修复完成并验证后回写实际采用的改动，以及它如何消除根因。

## 记录 Bug

当用户要记录一个缺陷，或明确要求修复一个尚未建账的问题时：

1. 读取 `bug/index.md`，确认没有同一问题的未修复记录。
2. 分配下一个 `BUG_NNNN`，创建详情文件。
3. 根据已知事实写“现象”；能够稳定复现时同步写“测试方式”。
4. “根因”和“解决方法”保持空白；尚不能稳定验证时，“测试方式”也保持空白。
5. 把 Bug 加入 `bug/index.md` 的“待修复”。

退出检查：详情文件和 index 行都存在；现象写的是可观察事实；未确认内容没有被写成结论。

## 修复 Bug

1. 读取 `BUG_NNNN.md` 和 `bug/index.md` 对应行。
2. 建立或校准“测试方式”，确认它能稳定暴露当前问题。
3. 收集足以解释问题的证据，定位根因后写入“根因”。
4. 围绕根因实施修复，并用同一测试方式验证。
5. 验证通过后回写“解决方法”，再把 index 行从“待修复”移到“已修复”，填写结束日期。

根因未确认、修复未完成或测试未通过时，Bug 继续留在“待修复”；报告当前证据和阻塞，不编造结论，不写解决方法。

退出检查：根因有证据支撑；测试方式可重复执行；解决方法与实际改动一致；测试证明现象已消失；index 行最终位于“已修复”。

## 边界

- Bug 只承载缺陷事实和修复结果，不补规格、AC、优先级、严重级别或实施计划。
- 需要规划独立交付时另建 TODO；Bug 本身不因修复工作变复杂而自动升级。
- Git 行为遵守 `CLAUDE.md`，Bug 修复完成不自动构成 commit 或 push 授权。
