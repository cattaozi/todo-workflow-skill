# skills/project —— 我怎么初始化工作间和挂载外部资源

> `projects/` 是 tf3 的工作间：放一套账本和记忆。外部项目、源码仓、资料目录通过软链挂到 `projects/links/`。壳仓不收编任何 `projects/` 生成物。

## 什么时候读我

当任务涉及下面情况时读我：

- 第一次把一组外部目录交给 tf3 管。
- 需要把多个文件目录地址软链引入。
- `projects/` 缺失、结构不完整或边界不清。
- 需要判断某个文件属于壳仓、工作间，还是外部资源。
- 需要创建、修正、解释工作间结构。

## 我负责什么

- 定义引入项目时生成的最小工作间结构。
- 创建工作间目录和入口文件。
- 将用户提供的外部目录软链到 `projects/links/`。
- 维护三层边界：壳仓、工作间、外部资源。
- 说明各目录职责。

## 我不负责什么

- 不替代 `todo / prd / explore` 的具体推进规则。
- 不定义 `todo / epic / prd / explore` 的账本字段和表结构。
- 不把 `canon.md` 扩展成独立工作流。
- 不规定外部项目怎么写代码、测试、构建或部署。
- 不把 `projects/` 生成物提交进 tf3 壳仓。

## 生成结构

```text
projects/
├── .gitignore
├── canon.md
├── inbox/
├── links/
│   ├── app-a -> <外部目录 A>
│   └── app-b -> <外部目录 B>
├── todo/index.md
├── epic/index.md
├── prd/index.md
├── explore/index.md
├── lesson/
├── log/
└── room/
```

这些文件和目录都在引入外部资源时动态生成，不属于 tf3 壳仓资产；壳仓只保留 `projects/.gitkeep` 作为挂载点。

职责：

- `links/`：外部资源软链区；用户提供多个目录地址时，统一软链到这里。
- `inbox/`：用户扔料的临时落点；归类后清空，不当长期仓库。
- `canon.md`：我对 `links/` 下外部资源的记忆本；只记录代码里查不到、用户告诉过、或需要长期记住的人话信息。
- `todo/`：散户 TODO 账本目录；具体格式由 `skills/todo.md` 定义。
- `epic/`：Epic 账本目录；具体格式由 `skills/todo.md` 定义。
- `prd/`：需求账本目录；具体格式由 `skills/prd.md` 定义。
- `explore/`：想法和预研账本目录；具体格式由 `skills/explore.md` 定义。
- `lesson/`：踩坑记录，一坑一条。
- `log/`：过程草稿和临时沉淀。
- `room/`：从账本渲染出的汇报，只给你看，我不读回。

## 入口文件内容来源

`project` 只负责编排初始化，不拥有各入口文件的内容格式。生成入口文件前，先读对应真相源：

- `canon.md`：本文件定义初始空壳；后续只写代码里查不到、用户告诉过、或需要长期记住的信息。
- `todo/index.md`：读 `skills/todo.md`。
- `epic/index.md`：读 `skills/todo.md`。
- `prd/index.md`：读 `skills/prd.md`。
- `explore/index.md`：读 `skills/explore.md`。

对应 skill 改了格式，初始化时跟随对应 skill；不在本文件复制字段。

### `canon.md` 记录什么

`canon.md` 是工作间的记忆本。它记录：

- tf3 管了哪些外部资源。
- 每个资源在人话层面是什么。
- 启停 / 运行这类你希望 tf3 记住的操作入口。
- 你告诉过 tf3 的背景、偏好、约束、历史决定。
- tf3 对这些项目的开放想法，但必须标明不是事实。

### `canon.md` 初始结构

```md
# Canon

## 外部资源

| 名称 | 路径 | 功能 | 一句话理解 |
|---|---|---|---|

## 启停方式

| 项目 | 启动方式 | 停止方式 | 备注 |
|---|---|---|---|

## 用户告诉我的事

## 我的想法

## 未确定
```

## 动作协议

### 初始化工作间

1. 若 `projects/` 已存在，先审计结构并只补缺失项；不存在再创建生成结构。
2. 创建 `projects/inbox/` 和 `projects/links/`。
3. 创建工作间 `.gitignore`，至少忽略：

```gitignore
links/
room/
.DS_Store
```

4. 创建 `canon.md` 空壳。
5. 按“入口文件内容来源”创建 `canon.md`、`todo/index.md`、`epic/index.md`、`prd/index.md`、`explore/index.md`。
6. 若工作间还不是 git 仓，询问是否初始化为独立 git 仓。

### 软链引入外部目录

当你提供一个或多个文件目录地址，并表达“软链引入 / 挂进来 / 交给 tf3 管”时，我按这个流程做：

1. 逐个检查路径是否存在、是否可读、是否为目录。
2. 默认用目录名生成 link 名；若 link 名冲突或目录名不适合，先给出建议映射等你确认。
3. 确保工作间已初始化。
4. 在 `projects/links/` 下创建软链：`<link-name> -> <外部目录绝对路径>`。
5. 不复制外部目录内容，不把外部资源提交进 tf3 壳仓或工作间仓。
6. 完成后报告成功项、跳过项和需要你拍板的冲突项。

### 修正工作间结构

当 `projects/` 已存在但结构不完整、软链冲突或边界不清时，我按这个流程做：

1. 对照“生成结构”审计当前 `projects/`。
2. 只补缺失的目录和入口文件，不重写已有账本。
3. 已有入口文件格式旧了时，先说明差异和风险，等你确认再改。
4. 软链冲突时列出当前指向和建议指向，等你拍板。
5. 发现外部资源被复制进工作间时，提醒并建议改成 `links/` 软链。

## 红线

- 壳仓只收 tf3 自己，不收任何 `projects/` 生成物。
- 若工作间启用独立 git，工作间仓只版本化 canon 和账本，不收 `links/` 和 `room/`。
- 外部资源由它自己的仓库或文件系统管理，不混入 tf3 账本仓。
- 初始化只建最小结构，不预造空 TODO、PRD、EXP 或 Epic。
- 不在本文件复制任何账本字段；字段变化只改对应 skill。
- `canon.md` 不记录能从外部资源中重新读取或推理出来的代码结构。
