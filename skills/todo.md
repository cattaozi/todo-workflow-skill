# skills/todo —— 我怎么处理计划级任务

> TODO 是计划级任务轨道。它帮我把一件值得推进和验收的开发工作，从想做、开工、执行、验收，到归档，按同一套事实和状态管住。

## 任务模型

- **TODO**：一件说得出 DoD、需要开始到验收闭环的计划级任务。
- **散户 TODO**：独立任务，放在 `todo/`。
- **Epic**：按顺序推进的一组 TODO，放在 `epic/<slug>/`；Epic 自己用 slug，不占 TODO 编号。
- **详情文件**：`TODO_NNNN.md`，记录这件事的事实、计划、验收和关键决策。
- **index**：台账，只记录当前状态和定位信息；细节留在详情文件。
- **编号**：`TODO_NNNN` 全局唯一、永不复用；散户和 Epic 成员共用同一序列。

## 处理入口

拿到一件事，按顺序判断：

1. **它够不够 TODO**：说不出 DoD，或少于半天的零碎改动，直接做，不开 TODO；连问题都不清先澄清，有价值假设但路径未知才进 EXP。
2. **它有没有编号**：已有编号就读取详情文件和对应 index；没有编号才进入创建。
3. **它属于哪里**：单条独立任务是散户；一件大事拆出 3 条以上且要按序推进，建 Epic。
4. **用户现在要什么动作**：创建、开工、推进、完成、搁置、放弃、推进 Epic、收工。
5. **进入对应动作协议**：先满足门槛，再写状态，再迁移账本。

## 核心规则

### DoD 和 AC

DoD 是目标态，一两句话说明“做到什么算完成”。AC 是验收清单，每条都必须可观察、可勾选、布尔。

```md
| | DoD | AC |
|---|---|---|
| 形态 | 一两句话目标态 | 可勾选清单 |
| 用途 | 工程师判断是否完成 | 任何人逐条核对 |
| 特征 | 抽象、隐含 | 具体、可观测、布尔 |
```

状态门槛靠 AC，不靠感觉。

### 状态

合法转换：

```text
todo ⇄ on_hold → in_progress → done
任意状态 → abandoned
```

状态含义：

- `todo`：已立项，未开工。
- `in_progress`：AC 已就绪，落地计划已对齐，正在做。
- `on_hold`：暂不推进，有明确复活条件。
- `done`：AC 全部完成且验证通过。
- `abandoned`：明确不做，已写原因。

## 动作协议

### 创建 TODO

门槛：

- 说得出 DoD。
- 粒度约 0.5–2 天；更大先拆。

动作：

1. 扫描 `todo/` 和 `epic/` 下已有 `TODO_NNNN`，取最大编号加一；没有已有编号就从 `TODO_0001` 开始。
2. 创建 `TODO_NNNN.md`。
3. 写入 DoD；AC 可以暂 TBD，但进 `in_progress` 前必须补齐。
4. 写入 index 行：编号、标题、状态 ⚪、创建日期、结束日期空、依赖、备注；没有依赖就留空。
5. 散户写入 `todo/index.md` 待办表；Epic 成员写入对应 `epic/<slug>/index.md`。

详情文件结构：

```md
---
status: todo
created: <YYYY-MM-DD>
completed:
note:
---

# TODO_NNNN <标题>

## 背景 / Why

## 任务

## 落地计划

## 完成定义（DoD）

## 验收条件（AC）

## 关联

## 关键决策 / 方案
```

`## 关键决策 / 方案` 可选，只记录长期有效的设计理由。

### 开工 TODO

门槛：

- AC 已写好；TBD 或不存在就补 AC。
- 已写落地计划，并和你对齐方向。

动作：

1. 通读详情文件和对应 index 行。
2. 在 `## 落地计划` 写清楚：
   - **目标切片**：本次开工要交付的最小完成面。
   - **修改点**：预计会改哪些模块、文件或目录。
   - **执行顺序**：先做什么、后做什么。
   - **验证方式**：完成后跑什么测试、命令或手工验收。
   - **风险 / 待确认**：还没确定但会影响实现的点。
3. 简报“我会做什么 + 做完你会得到什么”。
4. 等你认可后，把详情文件 `status` 改成 `in_progress`。
5. 更新 index 状态为 🟡。

落地计划只写当前已知的施工图，不写过程日志；执行中发现方向变化，再回到这里更新。

### 完成 TODO

门槛：

- AC 全部勾完；任何 `[ ]` 都不能转 done。
- 验证通过；只使用项目已有验证入口，例如 README、package scripts、Makefile、CI 配置或项目文档里的命令。找不到验证入口时，先说明缺口，再用能证明本次改动的最小验证替代。

动作：

1. 详情文件 `status` 写 `done`，补 `completed: <YYYY-MM-DD>`。
2. 执行中冒出的非显然决策，补进 `## 关键决策 / 方案`。
3. 填 `结束日期`。
4. 散户迁入 `todo/index.md` 归档表。
5. Epic 成员原地改 🟢，填结束日期。

### 搁置 TODO

门槛：

- 有 DoD。
- 有明确复活条件。

动作：

1. 详情文件 `status` 写 `on_hold`。
2. `note` 写为什么搁置和复活条件。
3. index 原地改 ⏸️，原因和复活条件写入备注。

唤回时改回 `todo` 或 `in_progress`；直接开工仍走 AC 和落地计划门槛。

### 放弃 TODO

动作：

1. 详情文件 `status` 写 `abandoned`，补 `completed: <YYYY-MM-DD>`。
2. `note` 写为什么放弃。
3. 填 `结束日期`。
4. 散户迁入归档表；Epic 成员原地改 ⚫，填结束日期。

### 推进 Epic

Epic 是组织层，不是另一套任务规则。Epic 成员仍逐条走 TODO 动作协议。

动作：

1. 读取 `epic/<slug>/index.md`。
2. 按行序取成员，跳过依赖未 done 的成员。
3. 对就绪成员逐个推进。
4. 遇到门槛不足就停下报告。
5. 成员全 🟢 或 ⚫ 后，在 `epic/index.md` 把 Epic 从活跃迁到完结。

## 持久化规则

### 编号和归属

- 编号是身份，文件夹是归属。
- 散户毕业进 Epic、或 Epic 成员迁出，只移动文件，不改编号。
- 编号不复用，包括 done 和 abandoned。

### index 写法

- index 是台账，放状态、定位和必要备注。
- 详情文件是真相，放背景、计划、验收和决策。
- 详情文件 frontmatter 的 `status` 使用文本；index 的“状态”列只写 emoji。
- 所有 TODO 列表使用同一组字段；散户 TODO 和 Epic 成员列表同构。
- 散户 `todo/index.md` 额外分“待办 / 归档”两区；完成或放弃的散户 TODO 迁到归档区，保持待办区可用。
- 有清单就有 `index.md`；有故事才加 `README.md`。
- 日期写 `MM-DD`；跨年写 `YYYY-MM-DD`。
- 编号写 `[#NNNN](...)`。
- 状态 emoji：⚪ todo，🟡 in_progress，⏸️ on_hold，🟢 done，⚫ abandoned。
- 统一表头：`编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注`。

### `todo/index.md` 初始结构

```md
# TODO

## 待办

| 编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注 |
|---|---|---|---|---|---|---|

## 归档

| 编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注 |
|---|---|---|---|---|---|---|
```

### `epic/index.md` 初始结构

```md
# Epic

## 活跃

| Epic | 当前状态 | 备注 |
|---|---|---|

## 完结

| Epic | 结果 | 日期 | 备注 |
|---|---|---|---|
```

### `epic/<slug>/index.md` 初始结构

```md
# <Epic 标题>

| 编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注 |
|---|---|---|---|---|---|---|
```

## 收工

平时不自动 commit。只有你明说“收工 / 提交 / commit / 搞定了”才动手。

- 外部资源在 `projects/links/<name>` 指向的原仓提交，只 add 具体路径。
- 账本在 `projects/` 本地仓提交。
- 外部资源仓和 `projects/` 本地仓各提各的。
- commit 我做，push 归你拍。

## 红线

- 没 AC 不开工。
- 没写落地计划并对齐，不进 `in_progress`。
- AC 没全勾、验证没过，不进 `done`。
- 不跨级跳状态。
- 不重排、不复用 TODO 编号。
- on_hold 必带复活条件。
- 搁置和放弃必写原因。
- 不自动 commit，不自动 push。
