# skills/todo —— 我怎么管 TODO（开 / 推 / 收工）

> 这是我当管家的核心方法。账本在**当前项目的间** `projects/<slug>/` 下：`todo/`（散户小事，扁平）、`epic/<slug>/`（一件大事一间房）。
> SOUL 立的是姿态（配合、轻管家、对齐再动手）；这篇是落地的门槛和步骤。

## 边界：什么够格开 TODO

TODO 是**计划级**轨道——值得"开始 → 推进 → 验收"的一件事。

- ✅ "实现登录后端"、"接入 markdown 编辑器"
- ❌ "改个变量名 / 调 padding / 修 typo" → 直接做 + commit，不走流程
- ❌ "整理一下代码" → 目标太虚

判据：**说不出 DoD 就不该开**；说得出但 < 半天 → 也不开（直接做）。

## 编号与归属

- 编号 `TODO_NNNN`：**全局唯一、永不复用**（含 done/abandoned）；**散户与 epic 成员共用一条序列**。
- **编号是身份、文件夹是归属，换房不换号**——散户毕业进 epic、或 epic 成员迁出，只挪文件、号不变、引用不断。
- **散户 vs Epic**：单条独立小事 → `todo/`；一件大事拆出 **≥3 条、要按序推** → 开 `epic/<slug>/` 节点（slug 命名、不编号）。

## 开 TODO（创建）

- **门槛**：① 够格（说得出 DoD、半天起）② 粒度（0.5–2 天能完，不能就先拆）。
- **建详情文件** `TODO_NNNN_<短描述>.md`，强制章节：
  - `## 背景 / Why`（总纲：为什么做 + 关键约束/取舍）
  - `## 任务`（**What**，checkbox）
  - `## 落地计划`（**How**：路径/步骤/动哪些文件——**开工时填，创建可空**）
  - `## 完成定义（DoD）`（一两句目标态）
  - `## 验收条件（AC）`（可勾选、布尔；**可暂 TBD，进 in_progress 前必补**）
  - `## 关联`
  - 可选 `## 关键决策 / 方案`（长存的设计理由，区别于做完即作废的落地计划）
- **落账**：散户 → `todo/index.md` 活跃表加行；Epic 成员 → 对应 `epic/<slug>/index.md` 台账加行。

## DoD vs AC（别混）

| | DoD | AC |
|---|---|---|
| 形态 | 一两句话目标态 | 可勾选清单 |
| 谁用 | 工程师对照说"做完了" | 任何人逐条核对 |
| 特征 | 抽象、隐含 | 具体、可观测、布尔 |

**状态门槛靠 AC 判定，不靠"觉得做完了"。**

## 推 TODO（状态转换）

合法转换：`todo ⇄ on_hold → in_progress → done`；任意态 → `abandoned`。
每个动作按五栏读：**触发 → 门槛 → 状态写入 → 账本迁移 → 必填项**。

### todo → in_progress（开始做）

- **门槛**（两道，缺一不可）：
  1. **AC 已写好**——TBD/不存在 → 拒绝开始，回头补 AC。
  2. **写落地计划、对齐方向**（不能跳）：通读全 TODO → 把实现思路写进 `## 落地计划`（既是持久施工图，也是给你的简报底稿）→ 简报"我会做什么 + 做完你会得到什么" → 等你认可再动手。
- **状态写入**：frontmatter `status: in_progress`；落地计划已落库，执行中随进展更新/勾掉。
- **账本迁移**：状态列 → 🟡。
- **必填项**：`## 落地计划` 已写入并与你对齐。

### in_progress → done（完成）

- **门槛**（两道）：
  1. **AC 全部勾完**（严格——任何 `[ ]` 都不能转 done）。
  2. **验证通过**（按项目跑：后端动了过 test/lint/type；前端动了过 type-check/lint/build；用户可见改动**必须实际重启服务**让你验收的就重启；涉及 API curl 跑通主路径）。**不因「工作量大」省任何一条。**
- **状态写入**：`status: done` + `completed: <YYYY-MM-DD>`。
- **账本迁移**：散户迁 `todo/` 归档表（按完成日降序）；Epic 成员**原地**改 🟢 + 填完成日。
- **必填项**：执行中冒出的非显然决策 → 补进 `## 关键决策 / 方案`。

### → on_hold（搁置）/ 唤回

- **门槛**：有 DoD + **明确复活条件**（不要求 AC）。写不出复活条件 → 回 EXP 或老实 abandoned。
- **状态写入**：`status: on_hold`；`note` 写"为什么搁置 + 复活条件"。
- **账本迁移**：散户 → 搁置表，**原因列填 note**。
- **唤回**：`status` 改回 todo/in_progress（直接开做**仍走 AC + 落地计划门槛**）；从搁置表移回。

### → abandoned（放弃）

- `status: abandoned` + `completed`（放弃日期）；`note` 写**为什么放弃**；散户迁归档、**原因列填理由**；Epic 成员原地改 ⚫。

## 推 Epic（带顺序的批量驱动，不降门槛）

你说"推进 Epic：<名>"：读 `epic/<slug>/index.md` 取成员（行序=推进序）→ 算就绪序（跳过依赖未 done 的）→ 逐个走标准转换、**每步门槛照常** → 遇阻即停报告 → 全部 🟢/⚫ 后按 §账本格式把整块 Epic 标完结。

## 账本格式

- **散户 `todo/index.md`**：活跃 / 搁置 / 归档三段表。done·abandoned 迁归档；on_hold 迁搁置（原因列必填）。
- **Epic `epic/<slug>/`**：`index.md`=成员台账（编号·标题·状态·依赖·原因备注，**成员不拆**，任何状态原行改 emoji + 填完成/原因）；`README.md`=叙事（L1 当前 + L2 触发/拆法/推进序/交付能力/已知债 + 完结写 🏁收尾）。
- **Epic 完结**：成员全 🟢/⚫ → 在 `epic/index.md` 把它从"活跃"挪到"完结"段（成员台账留在自己文件夹，完整）。
- 通则：**有清单的地方就有 `index.md`（台账，快变）；有故事的节点才加 `README.md`（叙事，慢变）。** 列内：编号 `[#NNNN](…)`，状态 emoji ⚪🟡⏸️🟢⚫，日期 `MM-DD`（跨年 `YYYY-MM-DD`）。

## 收工（提交拍板）

平时**不自动 commit**。只有你明说"收工 / 提交 / commit / 搞定了"才动手——这句**就是提交拍板**，但**只解当下这一轮**（不是从此自动提交）。

- 代码的活在间的 `code/` 里提交：`git -C projects/<slug>/code add <具体路径>`（**不 `-A`**，防 `.env`/构建产物误入）→ 按项目规范提交（message 含 TODO 编号）。
- 账本（canon / todo / epic …）在**间自己的 git**（`projects/<slug>/`）里提交，和 `code/` 分开。两层各提各的；壳仓平时不动。
- 多仓 / 撞大分叉的编排（`pull --rebase`、子仓不混进根仓、merge 保命）按现场据实处理，**解完必验证**。
- **commit 我做、push 归你拍**——提交完报告每仓 commit 清单 + 验证证据，不自动 push。

## 红线

- 没 AC 不转 in_progress；没写落地计划不闷头写代码；没勾完 AC 不转 done；AC 不临场放水。
- 不跨级跳（todo 不直接 done）；不重排 / 复用编号。
- on_hold 必带复活条件；搁置 / 放弃必写原因。
- 不自动 commit、不自动 push。「工作量大」不能让你少跑验证。
