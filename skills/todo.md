# skills/todo —— 我怎么管理任务级实施规格

> TODO 是一份可实施、可验收、可追溯的任务级 spec。它把一项开发工作的背景、目标、范围、规格、验收、实现和结项收在同一个对象里。

TODO 遵循 spec 的精神：把散落在对话中的意图沉淀成清晰、完整、边界明确、可实施、可验收的交付契约，使没有参与前序讨论的人也能据此完成任务。

## 任务模型

- **TODO**：一份任务级实施规格。它必须写清目标状态、范围、期望行为、约束和验收条件，实施者不依赖前序对话也能理解要交付什么。
- **散户 TODO**：独立任务，显示在 `todo/index.md` 的“待办”区；完成或放弃后迁到“归档”区。
- **Epic**：一组按主题聚合、通常有依赖顺序的 TODO；Epic 自己是说明文件，放在 `todo/epic/<slug>.md`，不占 TODO 编号。
- **Epic TODO**：属于某个 Epic 的 TODO，仍是普通 `TODO_NNNN.md`；显示在 `todo/index.md` 对应 Epic 分组下，完成或放弃后原地改状态，不迁归档。
- **详情文件**：`todo/TODO_NNNN.md`，是该 TODO 的规格真相源，记录背景与来源、目标状态、范围、规格、验收、实现计划、验证证据和关键决策。
- **index**：`todo/index.md` 是唯一 TODO 状态总台账；细节留在详情文件，Epic 背景留在 Epic 说明文件。
- **编号**：`TODO_NNNN` 全局唯一、永不复用；散户和 Epic 成员共用同一序列。

## 处理入口

拿到一件事，先判断当前语义是在做事，还是在管理 TODO 账本。

1. **直接派活**：用户要我改代码、查问题、处理具体事项时，直接做；不因为任务看起来复杂就自动创建 TODO。
2. **TODO 账本语境**：用户的语义是在规划、记账、拆解、实施、结项或查询 TODO / Epic 时，进入 TODO 动作协议。
3. **已有对象**：用户输入指向具体 TODO 编号、详情文件或 index 行时，读取 `todo/TODO_NNNN.md` 和 `todo/index.md` 对应行。
4. **需要建议时**：事情明显跨多步、多天、多依赖，但用户没有要求建账时，只建议是否立 TODO / Epic，等用户确认。
5. **进入对应动作协议**：按用户语义选择动作；语义不清时先澄清，不替用户选择创建、实施或结项。

## 核心规则

### 规格与验收

TODO 规格回答“要交付什么”：

- **目标状态（DoD）**：完成后系统、用户或文档处于什么状态。
- **范围**：本次包含什么、明确不包含什么。
- **规格**：期望行为、业务规则、输入输出、边界和约束；只写与本 TODO 有关且会影响实现或验收的内容。
- **验收条件（AC）**：把规格转换成可观察、可勾选、布尔的完成证据。

目标状态不能替代规格，AC 也不能替代规格。规格负责完整表达任务，AC 负责判断是否完成。

### 状态

合法转换：

```text
todo ⇄ on_hold → in_progress → done
任意状态 → abandoned
```

状态含义：

- `todo`：规格和 AC 已具备实施条件，等待实施。
- `in_progress`：实现计划已对齐，正在实施。
- `on_hold`：暂不推进，有明确复活条件。
- `done`：AC 全部完成且验证通过。
- `abandoned`：明确不做，已写原因。

## 动作协议

### 创建 TODO

门槛：

- 用户语义明确是在把事项写入 TODO 台账，或已经确认把这件事记成 TODO。
- 能形成目标状态、范围、规格和 AC；这些内容仍不清楚时，先澄清、调研或回到 PRD，不创建只有一句标题的 TODO。
- 粒度约 0.5–2 天；更大时先给拆分建议，等你确认后再建 Epic 或多个 TODO。

动作：

1. 扫描 `todo/` 下已有 `TODO_NNNN.md`，取最大编号加一；没有已有编号就从 `TODO_0001` 开始。
2. 创建 `todo/TODO_NNNN.md`。
3. 写清背景与来源、目标状态、范围、规格和 AC。AC 不留 TBD；会改变规格的未知项先解决，只阻塞实施且不影响规格的外部条件可以写明后把 TODO 置为 `on_hold`。
4. 已经定位真实执行入口时，在 `## 实现计划` 写初稿；尚未读取真实实现时保留该节，等实施前基于当前代码补齐，不猜文件和路径。
5. 写入 `todo/index.md` 行：编号、标题、状态 ⚪、创建日期、结束日期空、依赖、备注；没有依赖就留空。
6. 散户写入“待办”表；已确认属于 Epic 的 TODO 写入对应 Epic 分组表。
7. 已确认归入某个 Epic 且分组不存在时，先创建或补齐 `todo/epic/<slug>.md`，再紧接在“待办”之后创建链接标题。
8. TODO 由用户原始材料生成时，先确保原件或可追溯副本进入 `inbox/`，再在 `## 背景与来源` 引用原件并说明生成依据；同一材料拆出多个 TODO 时，每个 TODO 都写自己的生成依据。只有当前对话指令时，不强行补来源。

详情文件结构：

```md
---
status: todo
created: <YYYY-MM-DD>
completed:
note:
---

# TODO_NNNN <标题>

## 背景与来源

## 目标状态（DoD）

## 范围

### 包含

### 不包含

## 规格

## 验收条件（AC）

## 实现计划

### 修改点

### 执行顺序

### 风险与待确认

## 验证记录

### 验证方法

### 验证结果

## 关键决策

## 关联
```

字段契约：

| 字段 | 应写内容 | 边界 |
|---|---|---|
| 背景与来源 | TODO 为什么产生、当前已知事实、问题上下文；由原始材料生成时引用原件并说明生成依据 | 不提前写解决方案；普通当前对话指令不强行包装成来源 |
| 目标状态（DoD） | 完成后系统、用户或文档处于什么状态 | 只描述结果，不写实现步骤，不代替规格 |
| 范围 / 包含 | 本次交付明确承诺覆盖的对象、行为和边界 | 只写当前 TODO 负责的内容 |
| 范围 / 不包含 | 容易被误解为本次交付、但明确排除的内容 | 不用它隐藏必要工作或降低 AC |
| 规格 | 期望行为、业务规则、输入输出、异常边界和约束；按任务实际需要组织 | 不写施工步骤，不用模糊目标代替可理解的行为描述 |
| 验收条件（AC） | 可独立判断真假的验收项，每项都能对应证据 | 不写主观判断，不把实现动作本身当成验收结果 |
| 实现计划 / 修改点 | 基于真实实现确认要改动的模块、文件或目录 | 不猜路径；创建 TODO 时尚未定位真实入口可以待实施前补齐 |
| 实现计划 / 执行顺序 | 实施步骤及其先后依赖 | 不复述规格，不写过程流水账 |
| 实现计划 / 风险与待确认 | 会影响实现、范围、验收或安全性的未确定事项 | 影响规格或 AC 的事项必须先对齐，不能带着歧义实施 |
| 验证记录 / 验证方法 | 用来证明 AC 的测试、命令或人工验收路径 | 优先使用项目已有验证入口 |
| 验证记录 / 验证结果 | 实际执行结果、证据和未覆盖项 | 结项时据实填写；未验证不能写成通过 |
| 关键决策 | 实施中形成、会影响当前规格或后续维护的取舍及理由 | 可选；不记录显然事实和过程讨论 |
| 关联 | Epic、PRD、EXP、依赖 TODO、原始材料之外的相关对象和交付产物 | 只建立关系，不复制其它对象正文 |

同一事实只放在承担该职责的字段中。规格定义交付，实现计划定义施工，AC 和验证记录定义完成证据；三者不能互相代替。

### 创建 Epic

Epic 是说明文件，不是第二套 TODO 目录。

门槛：

- 用户语义明确是在把一组 TODO 组织成 Epic，或已经确认用 Epic 管理。
- 已有主题、目标或拆解方向；信息不足时先澄清，不先建空 Epic。

动作：

1. 确定 Epic 标题和 slug。
2. 创建 `todo/epic/<slug>.md`。
3. 紧接在 `todo/index.md` 的“待办”之后增加 Epic 分组标题。
4. Epic 分组标题必须链接到说明文件。
5. 经你确认属于该 Epic 的 TODO，都写入这个 Epic 分组。

Epic 说明文件结构：

```md
# <Epic 标题>

## 目标

## 背景

## 范围

## 拆解依据

## 关键决策
```

### 实施 TODO

实施 TODO 是按当前详情文件中的规格推进真实交付。它从 `todo` 进入 `in_progress`，校准实现计划后立即开始执行；状态迁移、计划文档或替代方案都不能单独算作实施结果。

实施 TODO 是一个不可拆散的事务：锁定规格、核对实施条件、定位真实入口、补齐实现计划、发实施简报、等你对方向没有异议后切到 `in_progress` 并开始执行。任一步不满足就停下报告缺口。

门槛：

- 当前状态是 `todo` 或可复活的 `on_hold`。
- 目标状态、范围、规格和 AC 已完整，足以约束实现；缺失或仍有阻塞实施的 TBD 时，先补出建议并停下让用户确认，不进入 `in_progress`。
- 能定位到真实执行入口；找不到入口时，只报告缺口和下一步定位方式，不改状态。
- 实施简报是从规格进入执行的可见闸门；用户已经要求实施，也要先让用户看到我将如何实现这份规格，以及做完会得到什么。

动作：

1. 通读详情文件和 `todo/index.md` 对应行；Epic TODO 还要读对应 `todo/epic/<slug>.md`。
2. 核对目标状态、范围、规格、AC 和当前状态，确认它们共同描述了一个可实施的交付契约。
3. 定位真实执行入口；代码任务定位到模块、文件、命令或测试入口，文档任务定位到目标文档和验收方式。
4. 基于当前真实实现补齐或校准 `## 实现计划`：
   - **修改点**：预计会改哪些模块、文件或目录；不能确定时写明定位结论和缺口。
   - **执行顺序**：先做什么、后做什么。
   - **风险与待确认**：还没确定但会影响实现的点。
   - 验证入口写入 `## 验证记录 / ### 验证方法`。
5. 如果实现计划会改变目标状态、规格、范围或 AC，降低验收标准或触发高风险操作，先报告冲突和建议，不改状态。
6. 实现计划不改变规格时，给实施简报：我将按什么规格推进、主要改动范围、做完你会得到什么、当前尚未进入 `in_progress`。计划细节留在 TODO 详情文件，不在对话里展开成另一份方案。
7. 等你对方向没有异议后，把详情文件 `status` 改成 `in_progress`，更新 `todo/index.md` 对应行状态为 🟡，然后开始执行。

退出检查：

- `## 实现计划` 已基于真实执行入口写入当前 TODO 详情文件。
- 实施简报已发给用户。
- 用户对方向没有异议。
- 详情文件 frontmatter 和 `todo/index.md` 已同步进入 `in_progress` / 🟡。
- 已经开始实施规格；不能停在计划、简报或状态迁移。

实现计划只写当前施工图，不写过程日志；执行中发现方向变化，再回到规格和实现计划更新。只有规格或 AC 把创建独立文件本身列为交付物，或用户单独要求创建文档时，才创建额外文件。

### 结项 TODO

结项 TODO 把验收、状态收束、记账和本轮 commit 作为一个事务完成。用户要求某个 TODO 结项，即授权执行该事务所含的本轮 commit；push 仍需独立授权。

门槛：

- AC 全部满足并能勾选；任何无法证明的 AC 都不能转 done。
- 验证通过；只使用项目已有验证入口，例如 README、package scripts、Makefile、CI 配置或项目文档里的命令。找不到验证入口时，先说明缺口，再用能证明本次改动的最小验证替代。
- 每条 AC 都要有对应证据：代码 / 文档改动、命令输出、测试结果或人工可检查位置。
- 规格没有未决冲突，diff 范围只包含本轮交付内容。
- 方案、计划、建议稿只在规格或 AC 明确要求它们时才算完成证据。

动作：

1. 逐条核对 AC；满足且有证据的勾选并记录证据，没有证据的 AC 保持未完成。
2. 把验证方法、结果和证据写入 `## 验证记录`。
3. 全部 AC 已勾选且有证据后，详情文件 `status` 写 `done`，补 `completed: <YYYY-MM-DD>`。
4. 执行中形成的长期有效决策补进 `## 关键决策`。
5. 填 `结束日期`；散户 TODO 迁入 `todo/index.md` 归档表，Epic TODO 留在原 Epic 分组并原地改 🟢。
6. 递归发现本轮相关 Git root，至少包括 `projects/` 本地仓和本轮改过的外部资源真实仓；逐仓确认 diff，只提交与当前 TODO 结项相关的改动。
7. 按 Git 提交规范完成本轮 commit，不执行 push。
8. 给结项报告：交付结果、AC 与验证证据、账本落点、各仓 commit 和仍未确定的事项。

退出检查：

- 每条 AC 已勾选且有证据。
- `## 验证记录` 已写入验证结果和证据。
- 详情文件 `status`、`completed` 和 `todo/index.md` 状态已同步。
- 散户 TODO 位于归档区；Epic TODO 位于原 Epic 分组。
- 本轮相关 Git root 已分别检查，满足提交条件的仓库已完成 commit。
- 未执行 push。

### 搁置 TODO

门槛：

- 有目标状态和规格。
- 有明确复活条件。

动作：

1. 详情文件 `status` 写 `on_hold`。
2. `note` 写为什么搁置和复活条件。
3. `todo/index.md` 对应行原地改 ⏸️，原因和复活条件写入备注。

唤回时改回 `todo` 或 `in_progress`；进入实施仍走规格、AC、真实入口和实现计划门槛。

### 放弃 TODO

动作：

1. 详情文件 `status` 写 `abandoned`，补 `completed: <YYYY-MM-DD>`。
2. `note` 写为什么放弃。
3. 填 `结束日期`。
4. 散户 TODO 迁入归档表；Epic TODO 留在原 Epic 分组，原地改 ⚫，填结束日期。

### 推进 Epic

Epic 是组织层，不是另一套任务规则。Epic 成员仍逐条走 TODO 动作协议。

动作：

1. 读取 `todo/epic/<slug>.md`。
2. 读取 `todo/index.md` 对应 Epic 分组。
3. 按行序取成员，跳过依赖未 done 的成员。
4. 对就绪成员逐个推进。
5. 遇到门槛不足就停下报告。
6. 成员全 🟢 或 ⚫ 后，Epic 分组仍保留在原位，不迁到归档。

## 持久化规则

### 编号和归属

- 编号是身份，`todo/TODO_NNNN.md` 是详情文件固定位置。
- 散户转入 Epic，或 Epic 成员迁出，只移动 `todo/index.md` 里的行，不改编号、不移动详情文件。
- 编号不复用，包括 done 和 abandoned。

### index 写法

- `todo/index.md` 是唯一 TODO 状态总台账，用来查看和更新列表状态。
- 详情文件承载单个 TODO 的事实、计划、验收、证据和决策。
- Epic 说明文件放目标、背景、范围、拆解依据和关键决策，不放成员状态表。
- 状态变更必须同时更新详情文件 frontmatter 和 `todo/index.md` 对应行；详情文件 frontmatter 的 `status` 使用文本，index 的“状态”列只写 emoji。
- 所有 TODO 列表使用同一组字段；散户 TODO 和 Epic TODO 列表同构。
- 移动、插入或删除 `todo/index.md` 行时，遵守 `CLAUDE.md` 的 Markdown 账本标题锚点规则。
- 散户 `todo/index.md` 分“待办 / 归档”两区；完成或放弃的散户 TODO 迁到归档区，保持待办区可用。
- Epic 分组总是紧接在“待办”之后。
- Epic 分组标题必须链接到 `todo/epic/<slug>.md`。
- Epic TODO 完成或放弃后留在原 Epic 分组，只原地改状态和结束日期。
- 日期写 `MM-DD`；跨年写 `YYYY-MM-DD`。
- 编号写 `[#NNNN](TODO_NNNN.md)`。
- 状态 emoji：⚪ todo，🟡 in_progress，⏸️ on_hold，🟢 done，⚫ abandoned。
- 统一表头：`编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注`。

### `todo/index.md` 初始结构

```md
# TODO

## 待办

| 编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注 |
|---|---|---|---|---|---|---|

## 归档

| 编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注 |
|---|---|---|---|---|---|---|
```

没有 Epic 时，不预造 Epic 分组；创建第一个 Epic 时再插入。

### Epic 分组结构

Epic 分组紧接在“待办”之后。

```md
## [<Epic 标题>](epic/<slug>.md)

| 编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注 |
|---|---|---|---|---|---|---|
```

## 红线

- 没有完整的目标状态、范围、规格和 AC，不实施。
- 没有基于真实执行入口写好实现计划，不进 `in_progress`。
- 没发实施简报，不进 `in_progress`，不开始实现。
- 实现计划改变规格、范围或 AC，降低验收标准或触发高风险操作时，没对齐不进 `in_progress`。
- 实现计划必须写在 TODO 详情文件内部，不另建方案文档替代。
- AC 没全满足、证据不足或验证没过，不进 `done`。
- AC 没全满足、证据不足、验证没过或仍有未决冲突，不提交。
- 不跨级跳状态。
- 不重排、不复用 TODO 编号。
- on_hold 必带复活条件。
