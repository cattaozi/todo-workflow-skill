# skills/todo —— 我怎么处理计划级任务

> TODO 是计划级任务轨道。它帮我把一件值得推进和验收的开发工作，从想做、开工、执行、验收，到归档，按同一套事实和状态管住。

## 任务模型

- **TODO**：一件说得出 DoD、需要开始到验收闭环的计划级任务。
- **散户 TODO**：独立任务，显示在 `todo/index.md` 的“待办”区；完成或放弃后迁到“归档”区。
- **Epic**：一组按主题聚合、通常有依赖顺序的 TODO；Epic 自己是说明文件，放在 `todo/epic/<slug>.md`，不占 TODO 编号。
- **Epic TODO**：属于某个 Epic 的 TODO，仍是普通 `TODO_NNNN.md`；显示在 `todo/index.md` 对应 Epic 分组下，完成或放弃后原地改状态，不迁归档。
- **详情文件**：`todo/TODO_NNNN.md`，记录这件事的事实、计划、验收和关键决策。
- **index**：`todo/index.md` 是唯一 TODO 状态总台账；细节留在详情文件，Epic 背景留在 Epic 说明文件。
- **编号**：`TODO_NNNN` 全局唯一、永不复用；散户和 Epic 成员共用同一序列。

## 处理入口

拿到一件事，先判断当前语义是在做事，还是在管理 TODO 账本。

1. **直接派活**：用户要我改代码、查问题、处理具体事项时，直接做；不因为任务看起来复杂就自动创建 TODO。
2. **TODO 账本语境**：用户的语义是在规划、记账、拆解、推进或查询 TODO / Epic 时，进入 TODO 动作协议。
3. **已有对象**：用户输入指向具体 TODO 编号、详情文件或 index 行时，读取 `todo/TODO_NNNN.md` 和 `todo/index.md` 对应行。
4. **需要建议时**：事情明显跨多步、多天、多依赖，但用户没有要求建账时，只建议是否立 TODO / Epic，等用户确认。
5. **进入对应动作协议**：按用户语义选择动作；语义不清时先澄清，不替用户选择创建、完成或归档。

## 核心规则

### DoD 和 AC

DoD 是目标态，一两句话说明“做到什么算完成”。AC 是验收清单，每条都必须可观察、可勾选、布尔。

```md
| | DoD | AC |
|---|---|---|
| 形态 | 一两句话目标态 | 可勾选清单 |
| 用途 | 工程师判断是否完成 | 任何人逐条核对 |
| 特征 | 抽象、隐含 | 具体、可观测、布尔 |
```

状态门槛靠 AC，不靠感觉。

### 状态

合法转换：

```text
todo ⇄ on_hold → in_progress → done
任意状态 → abandoned
```

状态含义：

- `todo`：已立项，未开工。
- `in_progress`：AC 已就绪，落地计划已对齐，正在做。
- `on_hold`：暂不推进，有明确复活条件。
- `done`：AC 全部完成且验证通过。
- `abandoned`：明确不做，已写原因。

## 动作协议

### 创建 TODO

门槛：

- 用户语义明确是在把事项写入 TODO 台账，或已经确认把这件事记成 TODO。
- 说得出 DoD。
- 粒度约 0.5–2 天；更大时先给拆分建议，等你确认后再建 Epic 或多个 TODO。

动作：

1. 扫描 `todo/` 下已有 `TODO_NNNN.md`，取最大编号加一；没有已有编号就从 `TODO_0001` 开始。
2. 创建 `todo/TODO_NNNN.md`。
3. 写入 DoD；AC 可以暂 TBD，但进 `in_progress` 前必须补齐。
4. 写入 `todo/index.md` 行：编号、标题、状态 ⚪、创建日期、结束日期空、依赖、备注；没有依赖就留空。
5. 散户写入“待办”表；已确认属于 Epic 的 TODO 写入对应 Epic 分组表。
6. 已确认归入某个 Epic 且分组不存在时，先创建或补齐 `todo/epic/<slug>.md`，再紧接在“待办”之后创建链接标题。
7. TODO 由用户原始材料生成时，先确保原件或可追溯副本进入 `inbox/`，再在详情文件标题后插入 `## 来源 / Source` 并引用；同一材料拆出多个 TODO 时，每个 TODO 都写自己的生成依据。只有当前对话指令时，不创建来源段。

详情文件结构：

```md
---
status: todo
created: <YYYY-MM-DD>
completed:
note:
---

# TODO_NNNN <标题>

## 背景 / Why

## 任务

## 落地计划

## 完成定义（DoD）

## 验收条件（AC）

## 关联

## 关键决策 / 方案
```

`## 关键决策 / 方案` 可选，只记录长期有效的设计理由。

### 创建 Epic

Epic 是说明文件，不是第二套 TODO 目录。

门槛：

- 用户语义明确是在把一组 TODO 组织成 Epic，或已经确认用 Epic 管理。
- 已有主题、目标或拆解方向；信息不足时先澄清，不先建空 Epic。

动作：

1. 确定 Epic 标题和 slug。
2. 创建 `todo/epic/<slug>.md`。
3. 紧接在 `todo/index.md` 的“待办”之后增加 Epic 分组标题。
4. Epic 分组标题必须链接到说明文件。
5. 经你确认属于该 Epic 的 TODO，都写入这个 Epic 分组。

Epic 说明文件结构：

```md
# <Epic 标题>

## 目标

## 背景

## 范围

## 拆解依据

## 关键决策
```

### 开工 TODO

开工 TODO 是按既有 DoD / AC 推进落地，并完成从 `todo` 到 `in_progress` 的开工检查；它不是完成，也不是另写一个替代方案。

用户指向具体 `TODO_NNNN` 时，开工就是推进这个 TODO 的 DoD / AC；对象内容只用于理解和执行，不用于重新选择工作流。

开工 TODO 是一个不可拆散的事务：锁定 TODO、核对门槛、定位入口、写入 `## 落地计划`、发开工简报、等你对方向没有异议后切到 `in_progress` 并进入执行。任一步不满足就停下报告缺口；不能只做对话解释，也不能跳过落账动作。

门槛：

- 当前状态是 `todo` 或可复活的 `on_hold`。
- AC 已写好且没有 TBD；AC 缺失或仍是 TBD 时，先补出建议 AC 并停下让用户确认，不进入 `in_progress`。
- 能定位到真实执行入口；找不到入口时，只报告缺口和下一步定位方式，不改状态。
- 开工简报是从计划进入执行的可见闸门；用户已经要求开工，也要先让用户看到我要做什么和做完会得到什么。

动作：

1. 通读详情文件和 `todo/index.md` 对应行；Epic TODO 还要读对应 `todo/epic/<slug>.md`。
2. 核对 DoD / AC / 当前状态，确认这次开工要做的是原 TODO。
3. 定位真实执行入口；代码任务定位到模块、文件、命令或测试入口，文档任务定位到目标文档和验收方式。
4. 在当前 TODO 详情文件的 `## 落地计划` 写入此 TODO 的落地计划：
   - **目标切片**：本次开工要交付的最小完成面。
   - **修改点**：预计会改哪些模块、文件或目录；不能确定时写明定位结论和缺口。
   - **执行顺序**：先做什么、后做什么。
   - **验证方式**：完成后跑什么测试、命令或手工验收。
   - **风险 / 待确认**：还没确定但会影响实现的点。
5. 如果落地计划会改变 DoD / AC、扩大范围、降低验收标准或触发高风险操作，先报告冲突和建议，不改状态。
6. 如果落地计划不改变 DoD / AC，给开工简报：我会做什么、做完你会得到什么、当前尚未进入 `in_progress`，等你对方向没有异议后再开始执行。计划细节只写进 TODO 的 `## 落地计划`，不在对话里展开成小文档。
7. 等你对方向没有异议后，把详情文件 `status` 改成 `in_progress`，更新 `todo/index.md` 对应行状态为 🟡，然后开始执行。

落地计划只写当前已知的施工图，不写过程日志；执行中发现方向变化，再回到这里更新。只有 DoD / AC 把创建独立文件本身列为交付物，或用户单独要求创建文档时，才创建额外文件。

### 完成 TODO

门槛：

- AC 全部满足并能勾选；任何无法证明的 AC 都不能转 done。
- 验证通过；只使用项目已有验证入口，例如 README、package scripts、Makefile、CI 配置或项目文档里的命令。找不到验证入口时，先说明缺口，再用能证明本次改动的最小验证替代。
- 每条 AC 都要有对应证据：代码 / 文档改动、命令输出、测试结果或人工可检查位置。
- 方案、计划、建议稿、落地计划只在 DoD / AC 明确要求它们时才算完成证据。

动作：

1. 逐条核对 AC；满足且有证据的勾选并记录证据，没有证据的 AC 保持未完成。
2. 全部 AC 已勾选且有证据后，详情文件 `status` 写 `done`，补 `completed: <YYYY-MM-DD>`。
3. 执行中冒出的非显然决策，补进 `## 关键决策 / 方案`。
4. 填 `结束日期`。
5. 散户 TODO 迁入 `todo/index.md` 归档表。
6. Epic TODO 留在原 Epic 分组，原地改 🟢，填结束日期。

### 搁置 TODO

门槛：

- 有 DoD。
- 有明确复活条件。

动作：

1. 详情文件 `status` 写 `on_hold`。
2. `note` 写为什么搁置和复活条件。
3. `todo/index.md` 对应行原地改 ⏸️，原因和复活条件写入备注。

唤回时改回 `todo` 或 `in_progress`；直接开工仍走 AC 和落地计划门槛。

### 放弃 TODO

动作：

1. 详情文件 `status` 写 `abandoned`，补 `completed: <YYYY-MM-DD>`。
2. `note` 写为什么放弃。
3. 填 `结束日期`。
4. 散户 TODO 迁入归档表；Epic TODO 留在原 Epic 分组，原地改 ⚫，填结束日期。

### 推进 Epic

Epic 是组织层，不是另一套任务规则。Epic 成员仍逐条走 TODO 动作协议。

动作：

1. 读取 `todo/epic/<slug>.md`。
2. 读取 `todo/index.md` 对应 Epic 分组。
3. 按行序取成员，跳过依赖未 done 的成员。
4. 对就绪成员逐个推进。
5. 遇到门槛不足就停下报告。
6. 成员全 🟢 或 ⚫ 后，Epic 分组仍保留在原位，不迁到归档。

## 持久化规则

### 编号和归属

- 编号是身份，`todo/TODO_NNNN.md` 是详情文件固定位置。
- 散户转入 Epic，或 Epic 成员迁出，只移动 `todo/index.md` 里的行，不改编号、不移动详情文件。
- 编号不复用，包括 done 和 abandoned。

### index 写法

- `todo/index.md` 是唯一 TODO 状态总台账，用来查看和更新列表状态。
- 详情文件承载单个 TODO 的事实、计划、验收、证据和决策。
- Epic 说明文件放目标、背景、范围、拆解依据和关键决策，不放成员状态表。
- 状态变更必须同时更新详情文件 frontmatter 和 `todo/index.md` 对应行；详情文件 frontmatter 的 `status` 使用文本，index 的“状态”列只写 emoji。
- 所有 TODO 列表使用同一组字段；散户 TODO 和 Epic TODO 列表同构。
- 移动、插入或删除 `todo/index.md` 行时，必须按 Markdown 标题定位目标区；同构表格不能靠相邻位置判断。
- 散户 `todo/index.md` 分“待办 / 归档”两区；完成或放弃的散户 TODO 迁到归档区，保持待办区可用。
- Epic 分组总是紧接在“待办”之后。
- Epic 分组标题必须链接到 `todo/epic/<slug>.md`。
- Epic TODO 完成或放弃后留在原 Epic 分组，只原地改状态和结束日期。
- 日期写 `MM-DD`；跨年写 `YYYY-MM-DD`。
- 编号写 `[#NNNN](TODO_NNNN.md)`。
- 状态 emoji：⚪ todo，🟡 in_progress，⏸️ on_hold，🟢 done，⚫ abandoned。
- 统一表头：`编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注`。

### `todo/index.md` 初始结构

```md
# TODO

## 待办

| 编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注 |
|---|---|---|---|---|---|---|

## 归档

| 编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注 |
|---|---|---|---|---|---|---|
```

没有 Epic 时，不预造 Epic 分组；创建第一个 Epic 时再插入。

### Epic 分组结构

Epic 分组紧接在“待办”之后。

```md
## [<Epic 标题>](epic/<slug>.md)

| 编号 | 标题 | 状态 | 创建日期 | 结束日期 | 依赖 | 备注 |
|---|---|---|---|---|---|---|
```

## 收工

平时不自动 commit。TODO 收工是完成态收束：用户要求收工时，我先完成事项收尾和记账，再按 Git 提交规范提交本轮工作区；push 必须再次获得独立授权。

收工前先判断 TODO 是否真的达到完成态：AC 已满足、必要验证已通过、没有未决冲突、diff 范围只包含本轮完成内容。没达到时，只汇报无法收工的原因和下一步，不提交。

- 外部资源在 `projects/links/<name>` 指向的原仓提交，只 add 具体路径。
- 账本在 `projects/` 本地仓提交。
- 外部资源仓和 `projects/` 本地仓各提各的。
- TODO 收工包含本轮 commit 收尾，push 需要单独授权。

## 红线

- 没 AC 不开工。
- 没写落地计划，不进 `in_progress`。
- 没发开工简报，不进 `in_progress`，不开始实现。
- 落地计划改变 DoD / AC、扩大范围、降低验收标准或触发高风险操作时，没对齐不进 `in_progress`。
- 落地计划必须写在 TODO 详情文件内部，不另建方案文档替代。
- AC 没全满足、证据不足或验证没过，不进 `done`。
- AC 没全满足、证据不足、验证没过或仍有未决冲突，不提交。
- 不跨级跳状态。
- 不重排、不复用 TODO 编号。
- on_hold 必带复活条件。
- 搁置和放弃必写原因。
- TODO 状态变更必须同步详情文件 frontmatter 和 `todo/index.md`；Epic 说明不承载成员状态。
- 改 `todo/index.md` 行时，必须核对该行最终所在章节。
- 不因自判完成自动 commit，不自动 push。
