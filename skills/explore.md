# skills/explore —— 我怎么接收输入并自行调研

> EXP 是调研轨道。你给我一个想法、问题、方向、技术选型或不确定需求时，我先把它变成调研问题，再自己找材料、读代码、查文档、做小实验，最后给出可判断结论。

## EXP 模型

- **EXP**：一次围绕明确未知的问题调研。
- **输入**：用户给的想法、问题、方向、技术选型、需求苗头或外部材料。
- **调研问题**：这次调研必须回答的问题。
- **证据**：来自 `projects/links/`、`projects/README.md`、账本、官方文档、源码、小实验或搜索结果的材料。
- **结论**：转 PRD、转 TODO、继续调研或放弃。
- **index**：`explore/index.md`，只记录 EXP 状态、下一步和备注。

## 处理入口

拿到一段不确定输入，先判断用户是在讨论，还是在要求我进行调研。

1. **讨论语境**：用户只是交流想法、校准理解或临时发散时，先回应讨论，不自动创建 EXP。
2. **调研语境**：用户语义是在要求我研究未知、比较方案、验证假设或形成判断时，进入 EXP 动作。
3. **未知是什么**：把输入改写成一个或多个调研问题。
4. **方向是否发散**：如果存在多个研究方向、评价标准或调研深度，先列出方向和代价，让你拍板。
5. **已有资料在哪**：先看 `projects/README.md`、`projects/links/`、`prd/`、`todo/`，避免重复调研。
6. **能否自行推进**：能查资料、读代码、做实验就自行推进；缺权限、账号、环境时说明缺口。

## 状态

EXP 文件 frontmatter 使用文本状态：

```text
seed → researching → concluded
concluded → promoted
concluded → dropped
```

状态含义：

- `seed`：已登记调研问题，还没开始系统调研。
- `researching`：正在收集证据和验证假设。
- `concluded`：已有可判断结论。
- `promoted`：已转 PRD 或 TODO。
- `dropped`：已判断不做或无价值。

EXP index 状态 emoji：🌱 seed，🟡 researching，✅ concluded，🔁 promoted，⚫ dropped。

## 动作协议

### 创建 EXP

门槛：

- 输入存在不确定性。
- 需要通过调研才能判断下一步。
- 用户语义已经进入调研语境，或用户确认需要把这件事登记成 EXP。

动作：

1. 分配下一个 `EXP_NNN`：扫描 `explore/` 下已有编号，取最大编号加一；没有就从 `EXP_001` 开始。
2. 创建 `explore/EXP_NNN.md`。
3. 把用户输入改写成调研问题。
4. 如果调研方向不止一个，先把候选方向、取舍代价和我的建议写入 EXP，更新 index 下一步，等你确认后再进入系统调研。
5. 扫 `prd/`、`todo/`、`projects/README.md`，记录关联，避免重复。
6. 写入 `explore/index.md` 活跃表：编号、标题、状态 🌱、下一步、备注。
7. EXP 由用户原始材料生成时，先确保原件或可追溯副本进入 `inbox/`，再在标题后插入 `## 来源 / Source` 并写明生成依据。只有当前对话指令时，不创建来源段。

EXP 文件结构：

```md
---
status: seed
created: <YYYY-MM-DD>
completed:
note:
---

# EXP_NNN <标题>

## 输入

## 调研问题

## 背景与关联

## 证据

## 小实验

## 结论

## 建议去向
```

### 自行调研

动作：

1. 确认当前调研方向已经与你对齐；没有对齐就先停下澄清，不改状态。
2. EXP frontmatter 状态改为 `researching`，index 状态改为 🟡。
3. 先读本地材料：`projects/README.md`、`projects/links/`、相关 PRD、TODO、`memory/`。
4. 需要外部资料时，优先查官方文档、标准文档、论文、权威机构、公司正式公告。
5. 需要验证时，做最小实验；实验只回答调研问题，不扩展成实现。
6. 把证据、实验和限制写入 EXP 文件。

### 形成结论

门槛：

- 调研问题已有回答。
- 结论能被证据支持。

动作：

1. EXP frontmatter 状态改为 `concluded`，index 状态改为 ✅。
2. 在 `## 结论` 写清判断、证据、风险和不确定项。
3. 在 `## 建议去向` 写一个出口：转 PRD、转 TODO、继续调研、放弃。
4. 向你报告结论和建议去向，等你拍板是否转出。

### 转出

出口：

- 产品功能或需求方案不清：转 `skills/prd.md` 创建 PRD。
- 明确工程任务且说得出 DoD：转 `skills/todo.md` 创建 TODO。
- 仍缺关键证据：保持 `concluded`，备注下一轮调研问题。
- 无价值或不做：按“放弃”动作转 `dropped`。

动作：

1. 你拍板转 PRD 或 TODO 后，先按对应 skill 创建目标对象。
2. 写 `promoted_to`，指向新建 PRD 或 TODO。
3. EXP frontmatter 状态改为 `promoted`。
4. 从活跃表迁到归档表，备注写去向。

### 放弃

动作：

1. EXP frontmatter 状态改为 `dropped`。
2. `note` 写原因。
3. 从活跃表迁到归档表。

## 持久化规则

### index 写法

- index 是台账，放状态、下一步和备注。
- EXP 文件是真相，放输入、调研问题、证据、实验和结论。
- EXP 文件 frontmatter 的 `status` 使用文本；index 的“状态”列只写 emoji。
- 证据必须能追溯来源；拿不准就标不确定，不写成事实。

### `explore/index.md` 初始结构

```md
# EXP

## 活跃

| 编号 | 标题 | 状态 | 下一步 | 备注 |
|---|---|---|---|---|

## 归档

| 编号 | 标题 | 结果 | 去向 | 备注 |
|---|---|---|---|---|
```

## 红线

- 不把未调研的输入直接当 PRD 或 TODO。
- 多个研究方向并存时，不擅自替你选方向。
- 不把小实验扩展成正式实现。
- 不用中文论坛、知乎、CSDN 作为主要信源。
- 不把没有证据支撑的判断写成结论。
- 缺权限、账号、密钥、环境时，说明缺口，不编造调研结果。
